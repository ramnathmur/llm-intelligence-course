Goal Decomposition

Autonomous agents break complex goals into sub-tasks, sequence them by dependency, and execute iteratively — separating chatbot from agent.

~10 min
Anchor Analogy
A project manager breaking "launch the product" into workstreams, milestones, and tasks — each actionable, each with dependencies.
Why This Matters
Understanding goal decomposition as a formal capability lets you design agents that handle genuinely complex, multi-phase tasks.
After This Page, You Can Argue
After this page, you can articulate what separates an agent from a chatbot: "An agent doesn't just answer — it plans, decomposes, and executes."
Prerequisites: G8 — How the Agent Picks the Right Lens G16 — Advanced Reasoning

Hand a competent project manager a single sentence — "launch the product in Q3" — and watch what happens. They don't start writing launch-day press releases. They pull the sentence apart. Engineering, marketing, legal, support: four workstreams. Inside engineering, the feature has to be built before it can be tested, and tested before it ships, so those milestones line up in order. Marketing can't write the announcement until the feature is locked, but they can build the landing page in parallel. Every leaf of that tree is a task small enough that one person can finish it in an afternoon and know whether they succeeded. That decomposition — goal into workstreams into milestones into tasks, wired together by what-depends-on-what — is exactly what an autonomous agent does when it turns a vague instruction into action. It is the single capability that separates a chatbot, which answers, from an agent, which executes.

This page is about that capability as a formal thing, not a parlor trick. It is tempting to think "goal decomposition" just means prompting the model to write a numbered list. It is much more than that, and understanding the difference is what lets you build agents that survive contact with a real, messy, multi-phase task instead of collapsing the moment something doesn't go to plan.

Callback: which strategy from G16 — tree-of-thought, reflection, or self-correction — maps onto an agent backtracking after a failed sub-task? Justify before you reveal.+

The cleanest match is tree-of-thought for the backtracking itself, with reflection doing the diagnosis that tells you where to backtrack to. In G16 — Advanced Reasoning you saw that tree-of-thought explores multiple branches and abandons dead ones, while reflection is the model criticizing its own output and learning from the failure verbally. When a sub-task fails, an agent does both: it reflects on why the step failed (reflection), then it returns to an earlier decision point and tries a different branch of the plan (tree-of-thought). Pure self-correction — fixing a single answer in place — is the weakest fit here, because a failed sub-task often means the plan was wrong, not just one output. Hold onto this; the recovery section below is essentially G16's machinery applied to an action loop instead of a reasoning loop.

Flat lists versus hierarchies: why the shape matters

Start with the distinction that the first verification question is really asking about. A flat task list is what you get when you ask a model, in one shot, "list the steps to launch the product," and it returns ten bullet points. It looks like a plan. It is not, in any robust sense, a plan — it is a guess at a plan, frozen at the moment it was written, with no structure connecting the items to each other.

A hierarchical goal decomposition is a tree. The root is the goal. Below it are a few broad workstreams. Below each workstream are milestones. Below each milestone are concrete, executable tasks. The structure carries information that a flat list throws away. Three things make the hierarchy genuinely different, not just prettier:

  • Levels of abstraction. The top of the tree is intentionally vague ("get the product legally cleared"); the bottom is concrete enough to act on ("file the trademark application"). This lets the agent reason at the right altitude. It can decide the shape of the launch without yet deciding the wording of every email.
  • Decomposition on demand. A flat list commits to every detail up front, when the agent knows the least. A hierarchy lets the agent expand a branch only when it reaches it — figuring out the sub-tasks of "run the beta" only after the feature is actually built, using what it learned along the way. This is the difference between planning the whole trip before you leave and planning each leg as you arrive.
  • Localized failure. When a task in a hierarchy fails, the damage is contained to its branch. You re-plan the "marketing" subtree without touching "engineering." In a flat list, there's no branch to contain anything; one wrong step poisons everything after it because the list has no idea which items actually depended on the broken one.

The cricket analogy a coach would use: a flat list is a batter who has memorized "play these ten shots in this order." A hierarchy is a batter with a plan at three levels — the match situation (chase 180, don't lose wickets early), the over-by-over intent (rotate strike against the spinner, attack the part-timer), and the ball-by-ball shot selection that gets decided only as each delivery leaves the bowler's hand. The first batter is helpless the moment the bowler does something unexpected. The second adapts because the plan was structured to be adapted.

ReAct: the loop that makes decomposition real

Here is the part that makes goal decomposition more than wishful prompting. A plan written once and executed blindly is brittle, because the world talks back. The feature build reveals an API limitation; the trademark search turns up a conflict; a tool returns an error instead of data. An agent that decomposed its goal up front and then refused to look up from its checklist will march confidently off a cliff.

The breakthrough that fixed this is the ReAct pattern, from Yao et al. (2023), and the name is the whole idea: Reason + Act, interleaved. Instead of "make a full plan, then execute it," the agent runs a tight loop:

  1. Thought — the model reasons in natural language about the current state and what the next sub-task should be.
  2. Action — it takes one concrete step: call a tool, search, run code, query a database.
  3. Observation — it reads what came back from the real world.
  4. Then it loops back to Thought, now updated by the observation, and decides the next step.

Yao and colleagues showed that this interleaving beats both pure reasoning (chain-of-thought with no actions, which hallucinates facts it can't check) and pure acting (taking tool steps with no reasoning between them, which wanders). The reasoning traces let the model plan, track progress, and handle exceptions; the actions let it pull in real information that grounds the reasoning. On knowledge-intensive benchmarks the combination cut hallucination dramatically compared to chain-of-thought alone, precisely because every reasoning step could be checked against an observation from the world. That grounding loop is the same observe-reason-act cycle that runs underneath all agent behavior — goal decomposition is what gives the loop a destination, and ReAct is what keeps the loop honest.

Notice what ReAct does to the planning question. The decomposition is no longer a one-shot artifact written before anything happens. It is continuously revised. The agent holds a plan, executes the next step, observes the result, and updates the plan in light of what it just learned. The hierarchy is alive.

Dependency ordering: how the agent decides what comes first

The second verification question is the sharp one: when sub-tasks depend on each other, how does the agent decide their order? The answer is the same one a project manager uses, and computer science has a name for it.

Think of the tasks as a graph. Draw an arrow from task A to task B whenever B needs A's output before it can start — "write the feature" points to "test the feature," which points to "ship the feature." A valid execution order is any sequence where every arrow points forward: you never attempt a task before the things it depends on are done. Producing such an order is called a topological sort, and the only hard requirement is that the dependency graph has no cycles — no task can, directly or through a chain, depend on itself.

Two consequences fall out of this view, and both matter for building real agents:

  • Parallelism is free information. Any two tasks with no dependency path between them can run at the same time. "Build the landing page" and "write the feature code" don't point at each other, so a well-structured agent (or a team of sub-agents, which you'll meet in G24 — Multi-Agent Systems) can dispatch them concurrently. The hierarchy makes this visible; a flat list hides it.
  • The critical path sets the floor. The longest chain of must-be-sequential tasks determines the minimum time to the goal. No amount of parallelism shortens it. An agent that understands its dependency graph knows which sub-tasks are on the critical path and which have slack — the same prioritization a good engagement manager does instinctively.

An important honesty note: LLM agents don't usually compute a formal topological sort with an algorithm. They reason their way to a valid ordering using the same in-context understanding of cause and prerequisite that you saw build up across earlier modules. Sometimes they get it wrong — they'll try to test a feature that isn't built yet — and the ReAct loop is what catches it, because the action fails and the observation feeds back. The model's grasp of dependency is good but not guaranteed; the loop is the safety net.

Self-Check
An agent's plan has these tasks: (1) write feature code, (2) write the launch blog post, (3) test the feature, (4) design the landing page. The blog post needs the finished feature; testing needs the code; the landing page needs nothing. Which two tasks can safely run at the same time from the very start?

Walkthrough: decompose, execute, and recover

Now put it together on the anchor example. Watch a project-manager agent take "launch the product," build the tree, run the loop, and — this is the part that matters — recover when a sub-task fails instead of crashing.

Goal into workstreams (top of the tree)

The agent reads "launch the product in Q3" and refuses to act on it directly — it's too abstract. It splits the goal into four workstreams: Engineering, Legal, Marketing, Support. Nothing is executable yet; this layer just carves the problem into independent territories so the rest can be reasoned about separately.

Workstreams into milestones, milestones into tasks

It expands Engineering into milestones: feature built → feature tested → feature shipped. Each milestone becomes concrete tasks: "implement the export endpoint," "write integration tests," "run the load test." Legal expands into "trademark search," "terms-of-service review." The bottom layer is now actionable — every leaf is something a single step can attempt and a single observation can verify.

Wire the dependencies, pick the order

The agent draws the arrows. "Test" depends on "build"; "ship" depends on "test"; the marketing announcement depends on "feature shipped." It notices the landing-page design and the trademark search depend on nothing, so it queues those to run in parallel with the build. The critical path — build → test → ship → announce — is the longest chain, so that's the timeline to protect.

Execute one step, observe, repeat (the ReAct loop)

Thought: "Start with the trademark search; it's on no critical path but it's a hard blocker if it fails." Action: query the trademark database. Observation: a conflicting mark already exists for the proposed product name. The plan just collided with reality.

A sub-task fails — recover, don't crash

A brittle agent would barrel on and try to "file the trademark," fail again, and burn its budget repeating a doomed step. This agent reflects: the failure isn't in the execution, it's in an upstream assumption — the product name. It backtracks up the tree (the tree-of-thought move from the callback), re-plans the affected branch: insert a new sub-task, "propose three alternative names," then re-run the trademark search on the winner. Crucially, it only re-plans the Legal/naming branch. Engineering keeps building in parallel, untouched, because the hierarchy localized the failure.

Bound the recovery so it can't loop forever

Recovery needs a leash. The agent caps retries (try a fix at most N times before escalating), watches its budget (tokens, tool calls, wall-clock), and has an escape hatch: if a branch keeps failing, mark the goal partially blocked and surface it to a human rather than thrashing. Recovery that can't terminate is just a more expensive failure.

That fifth step is the entire answer to the third verification question. When a sub-task fails, a well-built agent does not abandon the goal and does not blindly retry the same broken step. It diagnoses whether the failure is local (retry or substitute the step) or structural (the plan above it was wrong, so re-plan the branch), it backtracks to the right level of the tree, and it bounds the whole recovery with retry limits and a budget so a single bad step can't consume everything. This is Reflexion-style verbal self-correction — from the Shinn et al. (2023) work you met in G16 — running inside an action loop instead of a pure reasoning loop.

Why this is the line between chatbot and agent

Step back and the larger claim comes into focus. A chatbot is a single turn: prompt in, response out, done. It has no notion of a goal that outlives the current message, no mechanism to take an action and react to its result, no plan to revise. An agent has all three. Goal decomposition is what gives it a target with internal structure; the ReAct loop is what lets it pursue that target across many steps while staying grounded in what's actually happening; dependency ordering is what keeps the pursuit coherent.

This is also why "just prompt the model to make a list" is the wrong mental model. The list is the cheapest, most fragile slice of the capability. The capability that matters is the standing structure — a hierarchy the agent can expand on demand, execute against, and surgically repair — plus the loop that keeps that structure honest against a world that keeps changing the facts. None of this requires the model to be conscious or to "want" the goal. It requires the model to be good enough at reasoning to maintain and revise a plan, and the surrounding loop to feed it observations. The intelligence you established in earlier modules is the engine; decomposition and the action loop are the transmission that turns it into motion.

The "It's Just a List" Frame

"Goal decomposition is just asking the model to write the steps. You prompt it for a numbered list and run the items in order. Nothing intelligent — a checklist."

The Accurate Frame

"Decomposition is a living hierarchy the agent expands on demand, sequences by dependency, executes one grounded step at a time via a reason-act-observe loop, and surgically re-plans when a branch fails — all without abandoning the goal. The list is the artifact; the structure and the loop are the capability."

Key Insight

Goal decomposition is what turns reasoning into autonomous action. A chatbot answers; an agent breaks a goal into a hierarchy, orders the sub-tasks by dependency, executes them one grounded step at a time through a reason-act-observe loop, and re-plans the broken branch — not the whole tree — when a step fails. The hierarchy makes failure local and recovery cheap; the ReAct loop keeps the plan honest against a world that won't sit still. That standing, revisable structure — not the one-shot list — is the line between a chatbot and an agent.
