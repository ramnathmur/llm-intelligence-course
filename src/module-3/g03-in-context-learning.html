In-Context Learning

LLMs adapt their behavior from examples in the prompt without any training — runtime reasoning, not memorization retrieval.

~12 min
Anchor Analogy
An experienced consultant who reads your company's memo format from three examples and immediately produces a fourth.
Why This Matters
Directly answers "how will the AI know what lens to apply?" — the lens is in the prompt context; the model reasons from it.
After This Page, You Can Argue
After this page, you can demonstrate a capability no lookup table has: "Give it three examples of a task it's never seen — it learns the pattern on the spot, no retraining."
Prerequisites: G1 — What Weights Actually Encode G2 — Transformer Attention

You hire a senior consultant on a Friday. Monday morning you hand her three of your firm's old client memos — not a style guide, not a training course, just three finished documents — and say "we need a fourth, for the Henderson account, by noon." She reads the three. She notices the house format: the one-line situation summary up top, the three-bullet recommendation, the risk caveat in italics at the end. By 11:45 she hands you a Henderson memo that looks like it belongs in the same folder. Nobody retrained her. Nobody rewired her brain over the weekend. She inferred the pattern from the examples and applied it on the spot.

That is in-context learning, and it is one of the strangest, most consequential things a large language model does. You put a few examples of a task into the prompt — a task the model may never have seen in exactly that form — and the model figures out the pattern and continues it correctly. No gradient descent. No weight changes. No new training run. The "learning" happens entirely inside a single forward pass, using only what is sitting in the prompt right now. The skeptic who says "it just retrieves what it memorized" has a real problem here, because there is nothing to retrieve. The model is reasoning from the examples in front of it.

Callback: which mechanism from G2 lets a later token "see" and copy the structure of earlier in-prompt examples? Name it before revealing.+

Attention — specifically the pattern researchers call an induction head. As you saw in G2 — Transformer Attention, attention lets any token route information from any earlier token based on relevance, not just proximity. An induction head is a learned attention circuit that does something very specific: it looks back for a place where the current pattern appeared before, then copies what came next. If the prompt already showed "apple → fruit" and now says "apple →", the induction head finds the earlier "apple", looks at the token that followed it, and predicts "fruit". That copy-the-continuation move, generalized, is the mechanical backbone of in-context learning. Hold onto it — we come back to it below.

What "learning without learning" actually means

The word "learning" is doing two completely different jobs in machine learning, and conflating them is the source of most confusion about ICL. Pull them apart and the whole thing snaps into focus.

The first kind of learning is training. This is the slow, expensive process from earlier modules: the model sees billions of examples, computes how wrong it was, and adjusts its weights a hair at a time. The result is baked into the model permanently. As you saw in G1 — What Weights Actually Encode, this is where the model's knowledge physically lives — in the numbers that make up the network. Training is how the consultant got her education: years of cases, slowly shaping her judgment.

The second kind is in-context learning, and it changes nothing permanent at all. The weights are frozen. After the conversation ends, the model remembers nothing — the next user gets the exact same model, untouched. What changed was not the model but the model's input. The examples in the prompt steered the frozen network toward a particular behavior for the duration of that one response. It is the difference between the consultant's education (training) and the consultant reading your three memos this morning (in-context learning). One reshaped who she is; the other just told her what you need right now.

This distinction is the cleanest answer to the second verification question — how ICL differs from fine-tuning. Fine-tuning is real training: you take a base model and run more gradient updates on it using a curated dataset, and you walk away with a new set of weights. It is permanent, it costs compute and engineering, and it changes the model for everyone who uses that fine-tuned copy. In-context learning costs nothing but prompt space, changes no weights, and applies only to the single request that contains the examples. Fine-tuning is sending the consultant back to school for a semester on tax law. In-context learning is dropping three tax memos on her desk before a meeting. When you need the model to behave differently across millions of future calls, you fine-tune. When you need it to follow a pattern for this one task, you show it examples in the prompt. The table below makes the contrast concrete.

Fine-Tuning

Runs gradient updates on a curated dataset. Changes the weights permanently. Costs compute and engineering time. Affects every future call to that model copy. Use it when you need durable new behavior at scale — the consultant goes back to school.

In-Context Learning

Puts examples in the prompt. Changes no weights at all. Costs only context-window space. Affects only the single request that contains the examples. Use it when you need a pattern followed right now — the consultant reads three memos before the meeting.

Watch it happen: a task the model has never seen

The first verification question is the heart of this page: why can a model solve a task format it has never seen, given only three examples? The cleanest way to answer is to invent a task that cannot possibly be in any training corpus, show the model three examples, and watch it produce the fourth. Let's define a deliberately made-up format — call it a "glorp" — and walk the model through it.

Example one establishes a possible rule

We feed the model: cat → tac-3. There is no such operation anywhere on the internet; we just made it up. From one example the model can't pin down the rule — maybe it's reverse-the-word plus the length? Reverse plus letter count? It holds several hypotheses open at once.

Example two narrows the hypotheses

Next: bird → drib-4. Now the pattern tightens. "bird" reversed is "drib", and 4 is the number of letters. The reverse-plus-length hypothesis survives; the others mostly die. The model is doing hypothesis elimination inside the prompt, exactly the way the consultant ruled out "maybe they always use bullet points" after seeing a memo that used a numbered list.

Example three confirms the rule

Third: hello → olleh-5. "hello" reversed is "olleh", length 5. Three for three. The rule is now nailed down: reverse the word, append a hyphen, append the letter count. Notice no human ever stated this rule. The model induced it purely from the examples sitting in its context.

The model produces the fourth with no weight update

We give it table → and the model returns elbat-5. "table" reversed is "elbat", length 5. Correct — on a transformation that did not exist until two minutes ago. Crucially, every weight in the network is identical to what it was before we typed the first example. Nothing was trained. The pattern was inferred and applied in a single forward pass.

How the machinery pulls it off

This is the induction head from the callback, doing its job. Attention heads scan back through the prompt, latch onto the structure of the earlier "word → transformed" pairs, and route that structure forward to shape the prediction for the new word. The model isn't retrieving "elbat-5" from memory — it cannot have, because the task is invented. It is composing the answer by applying a pattern it extracted seconds ago from three examples. That is reasoning at runtime, not lookup.

Put plainly: the model can solve a never-seen task format because the examples in the prompt supply the rule, and the model's attention machinery is general enough to extract that rule and apply it on the fly. The weights don't encode the glorp task — they encode the far more powerful ability to infer tasks from examples. Brown and colleagues, in the GPT-3 paper that put this phenomenon on the map (Brown et al., 2020), called this "few-shot learning": give the model a handful of demonstrations in the prompt and it performs the task, often nearly as well as a model that was explicitly fine-tuned for it. Their central finding was that this ability gets dramatically stronger as models get larger — small models barely do it, big ones do it well. ICL is something that emerges with scale.

Why this is reasoning, not a clever lookup

It is worth slowing down on the distinction, because it is the whole intellectual payload of the page. A lookup table maps fixed inputs to fixed outputs. To handle a new input, you must first add a row for it. The model does the opposite: it handles inputs it has no row for, by inferring the function that connects the example inputs to the example outputs and then evaluating that function on the new input. Inferring a function from examples and applying it to novel cases is, in any reasonable sense of the word, reasoning. We call it that without hesitation when a human does it from three memos. The same description fits the model.

There's a subtle and important research wrinkle here that keeps the claim honest. What exactly is the model "learning" from the examples? One view, supported by a line of work on small transformers, is that the examples let the model implement something like a tiny learning algorithm internally — effectively fitting a quick rule to the demonstrations, almost as if it were running a miniature training step in its head. Another view is more deflationary: the examples mostly tell the model which of its already-trained abilities to switch on — less "learning a new skill from three examples" and more "the examples locate a skill the model already has." The evidence suggests both effects are real and the balance depends on the task. The careful statement is this: the model is genuinely computing over the examples, not retrieving a stored answer — but how much brand-new capability is being constructed versus selected is still an open question, and we'll come back to exactly that tension in the Think Further box.

The ceiling: when more examples stop helping

If ICL were unlimited runtime learning, then more examples should always help — the way more relevant memos should make the consultant's draft better. But that is not what happens. Add a few examples and quality jumps. Add a few more and it keeps improving. Keep going and the curve flattens, and past a point performance can actually degrade. This ceiling is real, and it tells you something honest about how much "learning" is happening in the prompt.

Several forces cap it. The most basic is the context window — the model can only attend to a finite span of tokens at once, so you cannot pour in unlimited examples; eventually the earliest ones fall off the edge or get drowned out. Within that window, the model also weights examples unevenly: research has repeatedly found position effects, where demonstrations near the start and end of the prompt influence the answer more than those buried in the middle. And there is a quieter ceiling: the consultant who has fully grasped your memo format after three examples gains nothing from a fourth, fifth, or twentieth. Once the pattern is unambiguous, additional demonstrations are redundant at best and, if they're noisy or slightly inconsistent, actively confusing. The model is not accumulating skill the way training does; it is locking onto a pattern, and once locked, more data has diminishing — sometimes negative — returns.

That ceiling is exactly why ICL is "learning" in a real but bounded sense. Genuine training compounds: every additional example nudges the weights and the knowledge accumulates permanently. In-context learning doesn't compound — it adapts the frozen model's behavior for one task, and the adaptation tops out when the pattern is clear. It is potent and it is real reasoning, but it is reasoning within the abilities the weights already hold, not the construction of durable new ones. Recognizing the ceiling is what keeps the intelligence claim defensible rather than starry-eyed.

The payoff: this is how an agent picks its lens

Now the part that connects straight to building agents. The third verification question asks how you'd use ICL to tell a research agent which lens to apply — "critical" versus "exploratory", say. The answer is that you don't retrain the agent for each lens. You put the criteria in its context. This is precisely the mechanism behind G8 — How the Agent Picks the Right Lens: an agent "knows" which lens to use because the decision criteria sit in its prompt — in the system instructions, in a few worked examples, or in retrieved context — and its in-context reasoning applies those criteria to the task at hand.

Concretely, you'd write a system prompt that says what each lens means, then show one or two examples of a query being handled under each lens. When the agent meets a new query, it does the same thing the consultant did with the memos and the same thing the model did with the glorp task: it reads the examples, infers the rule (this kind of query gets the critical lens, that kind gets the exploratory one), and applies it — with no weight update, on the spot, per request. Change the examples and you change the behavior, instantly, no training run required. That flexibility is not a quirk of how agents are wired; it is in-context learning, applied to the choice of how to think rather than the choice of what word comes next. Every prompt you write to steer an agent's judgment is an ICL prompt.

Key Insight

In-context learning separates two meanings of "learning" that are usually confused. Training reshapes the weights permanently and compounds; in-context learning leaves the weights frozen and adapts behavior for a single request using only the examples in the prompt. Because you can invent a task that provably never existed, give three examples, and still get a correct fourth, ICL cannot be memory retrieval — it is the model inferring a rule and applying it at runtime. That same mechanism is how every prompt-steered agent "decides" how to behave: the criteria live in the context, and in-context reasoning applies them.
