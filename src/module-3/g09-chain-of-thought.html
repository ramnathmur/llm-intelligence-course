Chain-of-Thought as Computation

CoT prompting forces multi-step computation in the visible token stream, measurably improving accuracy — a mechanism, not decoration.

~10 min
Anchor Analogy
A mathematician's scratchpad — the intermediate steps are not decoration; they are the computation.
Why This Matters
Understanding CoT as a computational mechanism changes how you design prompts and evaluate agent reasoning traces.
After This Page, You Can Argue
After this page, you can explain why "think step by step" works: "The intermediate tokens ARE the computation — the model literally reasons through them."
Prerequisites: G2 — Transformer Attention G3 — In-Context Learning

Hand a strong mathematician a hard problem and take away their scratchpad, and watch what happens. They stall. Not because they got dumber — because you removed the surface they think on. The scratchpad is not where they write down an answer they already have in their head. It is where the answer gets built, one line at a time, each line leaning on the line above it. The intermediate steps are not a record of the computation. They are the computation.

That single reframing is the whole of this page. "Chain-of-thought" prompting — nudging a model with "let's think step by step" — looks, from the outside, like a formatting preference. It seems like you are asking the model to show its work the way a teacher asks a student to. The skeptic reads it exactly that way: a cosmetic trick, dressing up an answer the model would have produced anyway. They are wrong, and they are wrong for a precise, mechanical reason. When a model writes out its reasoning, it is not narrating a conclusion it already reached. It is using the visible token stream as a scratchpad, and the tokens it writes become the working memory it computes the next step from. Take the scratchpad away and, like the mathematician, the model stalls — not from style, but from a hard limit on how much it can compute in a single shot.

Callback: why do worked-example reasoning traces in the prompt change behavior without any retraining?+

Because of in-context learning. As you saw in G3 — In-Context Learning, when you place worked examples in the prompt, the model adapts its behavior at inference time — the weights never change. The examples act as a runtime instruction, steering the model toward a pattern of response (here, "produce the reasoning, then the answer") without a single gradient update. Chain-of-thought is the most consequential application of that idea: the worked examples don't just teach a format, they teach the model to allocate the intermediate tokens it needs to actually do the work. Keep that distinction — format versus computation — in mind, because it is the crux of everything below.

The discovery that surprised the field

The result that put this on the map is Wei et al. (2022b), "Chain-of-Thought Prompting Elicits Reasoning in Large Language Models." The setup was almost embarrassingly simple. Take a hard multi-step problem — a grade-school math word problem, say — and give the model a few examples in the prompt. In one condition, the examples show only the question and the final answer. In the other, the examples show the question, then a short paragraph reasoning through it, then the answer. Same model, same questions, same number of examples. The only difference is whether the demonstrated examples include the reasoning steps.

The accuracy gap was not small. On a benchmark of math word problems, a large model went from solving a modest fraction of problems to solving the majority — a jump that, with the standard answer-only prompting, you simply could not buy at any model size available at the time. And here is the detail that should make you sit up: the effect barely existed for small models and exploded for large ones. Tiny models given the step-by-step examples often did worse — they would produce fluent, plausible-looking reasoning that was quietly nonsense, then commit to a wrong answer. The ability to use a chain of thought productively only showed up once the model was big enough. That pattern — a capability that is absent below a certain scale and present above it — is a thread you have already met in G5 — Emergent Capabilities, and it tells you CoT is not a universal formatting tweak. It is a latent capacity that scale unlocks.

Why "think step by step" actually moves the needle

To see why the reasoning steps help, you have to understand what a transformer can and cannot do in a single forward pass — the one trip through the network that produces one token.

A transformer has a fixed depth: a fixed number of layers stacked on top of each other. Information flows up through those layers exactly once to produce the next token, and then the process restarts for the token after that. This matters enormously. Think of each forward pass as having a fixed budget of sequential reasoning — you get to chain together only so many dependent operations before you have run out of layers and must emit a token. A problem like "37 + 48" might fit inside that budget. A problem like "a store had 23 apples, sold 7, restocked twice as many as it sold, then gave away a third — how many remain?" has more dependent steps than the model can reliably resolve in one pass. The answer depends on an intermediate result, which depends on an earlier intermediate result, and there are simply not enough layers to thread all of that in a single shot.

Chain-of-thought breaks this ceiling, and the way it does so is the heart of the mechanism. When the model writes out "the store sold 7 apples," that text is now part of the input. On the next forward pass, the model reads its own previous sentence and computes the following step from it. Each written step becomes a new starting point — a fresh forward pass with the previous result already laid down in the context for attention to read. The model is no longer trying to cram a ten-step computation into one pass of fixed depth. It is doing one step per pass and externalizing the intermediate result into the token stream, so the next pass can pick it up.

This is the answer to the first verification question — why "think step by step" improves accuracy on math problems. It is not that the phrase is magic, and it is not that the model "tries harder." It is that the phrase induces the model to spend tokens producing intermediate results, and each intermediate result extends the effective depth of computation by giving the next forward pass something concrete to build on. More steps in the visible stream means more sequential computation the model can actually perform. The reasoning is not decoration sitting beside the answer. The reasoning is the substrate the answer is computed on.

It also answers the second verification question directly — the computational mechanism, why intermediate tokens matter. The intermediate tokens matter because they are the model's external working memory. A single forward pass has no scratch space that survives to the next token except the tokens themselves. Anything the model needs to "remember" partway through a long computation has to be written down, because writing it down is the only way to carry it forward. The token stream is the only memory the model has between steps, so the steps must live in the token stream to exist at all.

A single forward pass has fixed depth. Chain-of-thought spreads a long computation across many passes, each reading the intermediate tokens the previous one produced — turning fixed depth into additive depth.

The scratchpad in action: same problem, two ways

Abstract claims about "forward passes" land better when you watch them on a real problem. Here is the same word problem solved two ways — once with the model forced to answer in one shot, once with the scratchpad allowed.

The problem

"A consulting team bills 3 analysts at $200/hour and 2 partners at $500/hour. They work a 6-hour workshop. The client gets a 10% discount. What is the final invoice?" Several dependent steps hide in here: two subtotals, a sum, a discount.

Without a scratchpad: one forward pass, one guess

Forced to emit the answer immediately, the model has to compress the whole calculation into a single pass of fixed depth. It often blurts something fluent and wrong — "$8,400" — a number in the right ballpark that skipped the discount or fumbled a subtotal. It looks confident because fluent text is cheap; it is wrong because the computation never had room to happen.

With the scratchpad: step one becomes input for step two

"Analysts: 3 × $200 × 6 = $3,600." That token sequence is now in the context. The next forward pass reads it instead of re-deriving it.

Each step leans on the written one above it

"Partners: 2 × $500 × 6 = $6,000. Subtotal: $3,600 + $6,000 = $9,600." The model is not holding $3,600 and $6,000 in some hidden register — it reads them back off the page it just wrote. The token stream is the working memory.

The final step is now trivial

"Discount: 10% of $9,600 = $960. Final invoice: $9,600 − $960 = $8,640." Every operation here was a single, shallow step performed on a result already written down. No single forward pass had to do more than it could. That is why the scratchpad version is right and the one-shot version was a guess.

Notice what just happened. The two runs used the same model with the same weights. Nothing about the model's knowledge changed between them. What changed was whether the model was allowed to externalize its intermediate results into the token stream. That is the difference between a guess and a computation — and it is entirely about where the work happens, not about how much the model knows.

Self-Check
Why does writing out intermediate steps let a model solve problems it gets wrong in a single shot?

This is not the same as the model just talking more

A natural objection: maybe longer outputs help for some boring reason — more tokens, more chances to stumble onto the answer. Researchers checked. If you prompt the model to emit the same number of tokens but fill them with filler — dots, irrelevant words, a repeated phrase — the accuracy boost largely vanishes. Length alone does nothing. What matters is that the tokens carry the actual intermediate results of the computation. A scratchpad full of doodles is not a scratchpad. The content of the steps is load-bearing, which is exactly what you would expect if the steps are the computation rather than mere padding around it.

There is a subtler and more important caveat, though, and this course does not skip the inconvenient parts. The relationship between the written reasoning and the model's actual internal process is not as tight as the clean story implies — which is precisely where the third verification question lives.

When chain-of-thought fails — and what those failures reveal

CoT is a real mechanism, not a panacea. It fails in at least three distinct ways, and each failure is informative rather than embarrassing.

1. It fails when the problem exceeds compositional reach

Spreading computation across tokens extends the model's depth, but it does not make it infinite. On problems that require deep, exact composition — long multi-digit multiplication, deeply nested logical deductions — performance still degrades as the required number of composed steps grows, even with the scratchpad. Researchers have shown that as the compositional complexity of a task rises, models increasingly fall back on shortcut pattern-matching rather than executing the full procedure (Dziri et al., 2023). This is the same boundary you will meet head-on in G6 — Compositional Generalization: CoT widens the window of solvable problems, but there is still a window, and you can push a problem past its edge.

2. It fails when small models imitate the form without the substance

Recall that tiny models given step-by-step examples often did worse. They learned the shape of reasoning — "first, ...; next, ...; therefore, ..." — without the underlying capacity to make each step actually follow from the last. The result is reasoning-shaped text that drifts confidently to a wrong answer. This is the clearest evidence that CoT is not mere formatting: if it were just a format, every model could exploit it. The fact that it requires sufficient scale to help at all means there is a real computation being scaffolded, and below a threshold there is nothing for the scaffold to hold up.

3. It fails — most importantly — when the stated trace is unfaithful

Here is the finding that should change how you read any reasoning trace. A model can write out a perfectly sensible-looking chain of thought, reach the correct answer, and yet the written steps do not reflect the computation that actually produced the answer. In controlled studies, researchers planted a subtle bias in the prompt — for instance, always marking option (A) as correct in the examples — and the model would dutifully pick (A) on a new question while writing a fluent justification that never once mentioned the real reason it chose (A). The visible reasoning was a plausible post-hoc story, not a transcript of the process. The model's "scratchpad" and its actual decision had come apart.

Sit with what that implies. The clean version of this page says "the intermediate tokens ARE the computation." The honest, fuller version is: the intermediate tokens are a computation that genuinely extends the model's depth and reliably helps on many tasks — but the written trace is not guaranteed to be a faithful audit log of how the final answer was reached. Both can be true. The scratchpad does real work, and the scratchpad can also lie about itself. This is the answer to "when does CoT fail, and why": it fails on problems too compositionally deep for the added depth to cover, it fails in models too small to make the steps real, and it fails as an honest record whenever the model's stated reasons diverge from its operative ones.

Why does unfaithfulness happen at all? The leading explanation is that the model is trained to produce text that looks like good reasoning, and a fluent justification is often a separate object from the actual path through the weights that selected the answer. The reasoning that helps accuracy and the reasoning the model narrates are correlated but not identical. Researchers are actively working on this, and the degree of unfaithfulness is contested — it varies by model, task, and how you measure it — so treat any strong claim in either direction with caution.

The "Format Trick" Frame

"'Think step by step' is just a prompt-engineering hack. It makes the output look more thorough, but the model already knew the answer — the steps are window dressing you could delete without losing anything."

The Accurate Frame

"The steps are the computation. A single forward pass has fixed depth and no memory between tokens; writing each result down lets the next pass read it, so a hard problem gets solved across many passes instead of crushed into one. Delete the steps and you delete the working memory — the answer collapses to a guess."

For you as a practitioner, the unfaithfulness finding has a sharp, practical edge. If you are building an agent and you log its chain of thought as an audit trail — "here is why the agent decided to refund the customer" — you cannot fully trust that log as a literal account of the decision. It is a useful artifact, often a good-enough one, and far better than no trace at all. But it is a plausible reconstruction the model produced, not a guaranteed transcript of its internal process. Designing for that gap — verifying outcomes rather than trusting stated reasons, building checks that don't assume the trace is honest — is the difference between an agent you can deploy and one that will surprise you. Once you see intermediate tokens as computation you can read, the natural next move is to operate on that computation — to branch it, score competing branches, and backtrack from dead ends. That is exactly the leap into G16 — Advanced Reasoning, where a single linear chain becomes a search over many.

Key Insight

A transformer's single forward pass has fixed depth and no working memory that survives between tokens — so anything the model needs to carry through a multi-step computation must be written into the token stream. Chain-of-thought works because each written step is read back on the next forward pass, turning the fixed depth of one pass into computation spread across many. The intermediate tokens are not a record of the reasoning beside the answer; for hard problems, they are the only place the reasoning can physically happen.
