What Is a Model?

A model is a function that maps inputs to predictions by storing learned patterns as numbers.

~15 min
Anchor Analogy
A consultant who has read thousands of case studies and answers new questions by pattern-matching against prior experience — except the patterns are stored as numbers, not memories.
Why This Matters
Every AI tool you use is a model. Understanding what a model actually is dissolves the mystery and lets you reason about when they work and when they fail.
After This Page, You Can Argue
What a model is, what it stores, and why it generalises to unseen inputs.
Prerequisites: None — this is the entry point.

Picture a senior management consultant who has spent twenty years reading case studies. Show her a new client situation and she will not flip through a filing cabinet looking for an exact match. She will pattern-match: "This looks like the turnaround we ran in Johannesburg in 2018. Similar cost structure, similar market dynamics." Her advice comes from patterns compressed across thousands of engagements, not from memorising any one of them.

A machine learning model works the same way, except the patterns live as numbers rather than as human memory. The model reads a vast amount of examples, extracts the patterns that explain them, and stores those patterns as numbers. Ask it something new, and it pattern-matches against what it learned. It does not look up the answer — it derives it.

That one idea — patterns stored as numbers, applied to unseen inputs — is the entire conceptual core of machine learning. Everything else is implementation detail. This page builds that core carefully so you have a foundation to stand on for every page that follows.

The two systems you already know

Before defining a model precisely, it helps to contrast it with the two alternatives you almost certainly already use: lookup tables and rule-based systems.

Lookup tables

A lookup table stores known answers for known questions. A tax table is a good example. You find your income bracket, read across to the applicable rate, apply it. The table was built by someone who enumerated every case in advance. It works perfectly — until you encounter a situation the table's author did not anticipate.

Databases and spreadsheet VLOOKUP functions are both lookup tables at heart. They retrieve. They do not infer.

Rule-based systems

Rule-based systems replace the table with explicit logic. An early fraud detection system might say: "Flag any transaction over $10,000 that occurs in a country different from the card's home country." A human expert wrote these rules. The system applies them mechanically.

Rule-based systems are powerful when experts can articulate the logic precisely. They fail in two situations. First, when the space of inputs is too large for any set of rules to cover. Second, when the correct rule is tacit knowledge that experts cannot put into words — like recognising whether a sentence sounds natural, or whether a scan looks malignant.

Where models come in

A model is what you build when you cannot enumerate all the answers in advance and cannot write down all the rules. Instead of specifying the logic, you show the system many examples and let it discover the logic for itself. The discovered logic is stored as numbers. That collection of numbers is the model.

Key Distinction

Rule-based systems encode human knowledge as logic. Models encode statistical patterns as numbers. The difference determines what each can generalise to.

What a model actually is: a function

Strip away all the jargon and a model is a function. It takes inputs, applies a transformation, and produces outputs. That transformation is defined by a set of numbers called parameters or weights.

Consider the simplest possible model: a straight line. You want to predict next month's sales from this month's marketing spend. The model is:

predicted_sales = weight × marketing_spend + bias

Two parameters: weight and bias. The weight controls how steeply sales respond to spend. The bias captures baseline sales with zero marketing. Given any value of marketing spend, the function produces a prediction.

That is a model. A frontier language model like GPT-4 is the same conceptual object, with hundreds of billions of parameters instead of two, and a much more complex function. But the core idea is identical: parameters define a mapping from inputs to outputs.

The anatomy of a model

Every model has three ingredients.

Architecture. The shape of the function — how inputs connect to outputs, how many parameters there are, what kind of transformations are applied. A straight line is one architecture. A neural network is another. The architecture is chosen by the engineer before training begins.

Parameters. The numbers that fill in the architecture. Before training, they are typically random. After training, they encode the patterns extracted from data. The parameters are what the model is, in a sense: swap the parameters and you have a completely different model, even with the same architecture.

A prediction rule. Given new inputs, the model runs the function using its learned parameters and produces an output. For a sales model, the output is a number. For an image classifier, it might be a label like "cat" or "dog." For a language model, it is a probability distribution over possible next tokens.

Rule-based system

A human writes logic: "If marketing spend > $50k AND quarter is Q4, predict sales > $1M." Logic is explicit. The system fails on any case the author did not anticipate. Hard to update when the world changes.

A model

Examples define the logic implicitly. The model discovers: sales and spend correlate in this season-adjusted way. Logic is stored as parameters. Updates by retraining on new data. Generalises to situations the author never saw.

What it means for a model to learn

Learning is the process of adjusting parameters until the model's predictions match the training data as closely as possible.

Return to the sales example. Start with random values for weight and bias. Feed in last year's actual marketing spend figures, one month at a time. For each month, the model produces a prediction. Compare the prediction to the actual sales figure. The gap is the error.

Now adjust the parameters in whatever direction reduces that error. Do this for every example. Repeat many times. Gradually, the parameters settle into values that make the model's predictions fit the data well. That process of parameter adjustment is called training. The algorithm that does the adjusting is called gradient descent.

After training, the parameters have changed from random noise into numbers that encode the relationship between marketing spend and sales. The model has learned that relationship from examples, rather than being told it by a rule.

Think first: if learning means fitting parameters to data, why not just memorise every training example? +

Memorisation would mean storing every (input, output) pair exactly. Two problems make this unworkable. First, storage: a language model trains on trillions of tokens. Storing them all verbatim would require more space than the model has parameters — the numbers simply do not fit. Second, and more importantly, memorisation does not generalise. If your sales model memorised last year's data, it would produce a perfect prediction for the months it saw and a completely blank stare for any new month. The whole point of a model is to extract patterns that apply to inputs it has never seen. Generalisation requires compression: the model must discard the specific examples and retain only the patterns they imply.

Why models generalise to unseen inputs

This is the question at the heart of machine learning. Training data is always finite. The world keeps producing new situations. Why should a model that learned from past examples do anything useful on future ones?

The answer is that useful patterns are persistent. If marketing spend predicted sales last year, it probably predicts sales this month too — not because you declared it to, but because the underlying mechanism (advertising drives awareness, awareness drives purchase) is real and stable. A model that discovers that pattern owns something that transfers across time and context.

The formal version of this idea is called the inductive bias of a model: the assumptions baked into its architecture about what kinds of patterns are worth looking for. A straight line assumes the relationship is, well, linear. A neural network assumes the relationship can be built up from many small non-linear transformations. Different architectures are better suited to different kinds of problems because their inductive biases align differently with the structure of the world.

The generalisation bargain

There is always a bargain involved. A model that fits training data perfectly — that can reproduce every quirk and noise spike in the examples it saw — has almost certainly memorised noise rather than pattern. It will perform poorly on new data. Statisticians call this overfitting.

A model that is too simple to fit the training data misses real patterns. Its predictions are wrong everywhere, not just on unseen inputs. That is underfitting.

Good training finds the middle: complex enough to capture genuine structure, simple enough to generalise rather than memorise. Every decision in machine learning — how many parameters, how long to train, how much data to use — is ultimately an attempt to navigate this bargain.

The amber line is validation error — how well the model performs on data it has never seen. It falls at first (the model is learning genuine patterns) and then rises (the model starts memorising noise). The sweet spot is where validation error is lowest. Everything in the practice of machine learning is an attempt to find and stay near that point.

What the parameters actually store

It is tempting to think of parameters as a filing cabinet, with each number storing one fact. They do not work that way.

In a linear model, the weight directly encodes the relationship between input and output. But in a neural network — the architecture behind modern AI systems — no single parameter corresponds to a single fact. Instead, facts and patterns are distributed across millions or billions of parameters. The same parameter participates in representing thousands of different concepts. Remove any single parameter and you will not excise one clean fact; you will degrade performance slightly across many tasks.

Think of it like the collective knowledge of a consulting team. No one person on the team owns "the Japan market strategy." That knowledge is distributed: one partner understands the regulatory environment, another knows the key clients, a third has done three similar engagements. The team can answer questions about Japan because the knowledge is encoded collectively. Ask any one person to carry the whole thing and you lose fidelity immediately.

This distributed encoding is why models generalise. If facts were stored individually, one example could corrupt one slot. Because facts are distributed, no single training example dominates any single parameter, and the system is forced to find configurations that are consistent across all examples simultaneously.

What a model cannot store

Distributed storage has one critical implication: models cannot update a single fact cleanly. If your linear sales model was trained before a global pandemic changed the relationship between marketing and sales, you cannot just edit the "marketing coefficient" parameter. The relationship is encoded across all parameters jointly. You retrain.

This is not a flaw — it is a consequence of the design. And it means that when a model gives you a confidently wrong answer about a recent event, it is not because it is lying. It is because it never saw evidence that would have shifted its distributed representation of that situation. The pattern it learned was real in its training data and simply no longer holds.

The three things a model must have

Every working model has three components, and a failure in any one of them breaks the system. Knowing which component failed is half the diagnostic work when a model misbehaves.

Data — examples from which to learn. No data, no model. Bad data, bad model. The single most reliable predictor of model quality is the quality of the training data, not the sophistication of the architecture.

An objective — a precise definition of what "doing well" means. For the sales predictor, doing well means minimising the gap between predicted and actual sales. The objective function makes that gap a number the optimiser can minimise. Without a clear objective, training has no direction.

An optimisation algorithm — the procedure that adjusts parameters to improve the objective. Gradient descent is the standard. It evaluates how each parameter affects the objective and nudges parameters in the direction that lowers the error. Run it long enough on enough data and parameters converge to values that make good predictions.

Self-Check
Your company wants to build a system that flags potentially fraudulent invoices. The fraud team says there are too many variations to write rules for all of them. Which approach is better suited?

The difference between a model and a rule-based system

This question comes up often in practice, and the answer is sharper than it first appears.

A rule-based system encodes logic that a human understood and expressed. Its behaviour is fully auditable: trace any decision back to the rule that produced it. Its failure mode is coverage: it can only handle situations its authors anticipated.

A model encodes logic that no human fully articulated. Its behaviour is partially opaque: the parameters encode the logic collectively, not as readable rules. Its failure mode is distribution shift: it handles situations that look like training data well, and situations that do not look like training data poorly.

Neither is universally better. A good engineering decision requires knowing which failure mode matters more for your situation. If the rules are few, stable, and well understood (tax calculations, physics simulations), rule-based systems are more transparent and easier to audit. If the patterns are complex, numerous, and implicit (fraud detection, image recognition, language), models are more capable.

Most real systems mix both. A model flags suspicious invoices; a rule then checks that the flagged invoice is above a threshold before triggering a human review. The model handles the complexity the rules cannot; the rules handle the policy constraints the model should not.

Why models fail

Understanding failure modes is the practical payoff of understanding what a model is.

Models fail when the world they encounter during deployment differs from the world they trained on. Call this distribution shift. If your sales model trained on pre-pandemic data, the relationship between spend and sales may have changed. If your fraud model trained on 2019 fraud patterns, new tactics it never saw will slip through.

Models also fail at the boundaries of their training data. Ask a model about a niche domain it saw little of during training and its predictions become unreliable. It does not know it has left the region where its patterns hold; it continues to produce predictions with the same apparent confidence. This is one of the most important things to understand about all current AI systems: they do not know what they do not know.

Finally, models fail when the objective they trained on does not match the outcome you actually care about. A recommendation model trained to maximise clicks may learn to recommend outrage-inducing content. A language model trained on a proxy for "helpfulness" may learn to produce text that sounds helpful rather than text that is helpful. Objective misspecification is the hidden source of many AI failures that look like intelligence problems but are really design problems.

Think Further

Every AI tool you use is a model. Pick one you use regularly. What was it probably trained on? What distribution shift might make it fail? What objective was it trained to maximise — and how closely does that objective match what you actually want?

Models in the ML Foundations curriculum

This course builds up from this foundation. The pages that follow examine specific types of models, specific training algorithms, and specific failure modes. Each page connects back to the three-ingredient view of models: data, objective, optimisation.

The most important thing to carry forward from this page is the function view. A model is not a database. It is not a chatbot with a personality. It is a function — a mathematical object that takes inputs, applies a transformation defined by learned parameters, and returns outputs. The wonder is not that such a function exists. The wonder is that you can learn one from data that is general enough to be useful on inputs the function has never seen.

Key Insight

A model is a function defined by learned parameters. Training adjusts those parameters until the function predicts well on known examples. Generalisation happens because training forces the model to compress patterns rather than memorise examples — and real patterns persist beyond the specific data they were extracted from.
