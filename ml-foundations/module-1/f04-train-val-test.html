Train / Validation / Test Splits

Three separate data pools that prevent a model from memorising instead of learning.

~15 min
Anchor Analogy
A law firm that prepares cases using known precedents (train), tests its arguments with a mock jury (validation), then faces the real jury on the day (test) — and never shows the mock-jury feedback to the real jury.
Why This Matters
Evaluating a model on the same data it trained on is like marking your own exam — you will overestimate performance. The three-split discipline is the foundation of honest model evaluation.
After This Page, You Can Argue
Why three splits are necessary, what each is used for, and what data leakage means.
Prerequisites: F01 — What Is a Model?

Imagine a law firm preparing to argue a major case. The senior partners spend weeks studying past rulings — precedents set in hundreds of similar disputes. They know those cases inside out. They can cite every judgment, anticipate every counter-argument, predict every objection. When they then rehearse in front of a mock jury drawn from within the firm itself, they perform brilliantly. Of course they do. The mock jury has heard all the same preparation sessions.

Then comes the real trial. The actual jury has seen none of the internal preparation. They bring fresh eyes, fresh questions, fresh scepticism. A firm that optimised entirely for its own mock jury — rather than the real-world challenge of persuading strangers — will find itself blindsided.

This is exactly the trap a machine learning model falls into if its evaluation is not designed carefully. A model that is tested on the same data it learned from will appear to perform far better than it actually does on new, unseen cases. The three-split discipline — dividing all available data into training, validation, and test sets — is the engineering practice that stops this from happening. It is one of the most important methodological ideas in all of applied machine learning.

The core problem: memorisation versus learning

A model learns by adjusting its internal parameters to minimise errors on the examples it sees during training. If the model is small and the problem is simple, this produces genuine generalisation — patterns discovered in the training data that also hold for new data. But if the model has enough capacity, it can do something subtler and more dangerous: it can memorise the training examples rather than extract transferable rules.

Memorisation looks like learning on the surface. The model produces correct outputs for every training example. Loss is very low. Accuracy is very high. But when you show it an example it has never seen, performance collapses. The model has not discovered the underlying pattern; it has stored the answers to a specific set of questions. This failure mode is called overfitting.

The clinical test for overfitting is simple in principle: evaluate performance on data the model has never seen during training. If performance is strong on training data but weak on fresh data, the model has overfit. If performance is similar on both, the model has learned something genuinely generalisable. This is why the training set must be held strictly separate from the evaluation data.

Think Further
A model memorising training data is not just a theoretical failure. Consider a fraud-detection model trained on historical transaction records. If it overfits, it learns to recognise the specific fraudsters from history — but misses entirely new fraud patterns that arrive after deployment. The cost of this gap is real.

Why two sets are not enough

The most intuitive fix is to split your data into two pools: train on one, evaluate on the other. This is better than nothing, but it creates a subtle problem the moment you start making modelling decisions based on the evaluation results.

Every time you evaluate on the held-out set and then change something — adjust a hyperparameter, try a different algorithm, add a new feature, change the data preprocessing — you are using information from that held-out set to guide your choices. You are, in effect, training on it indirectly. Over many rounds of "evaluate, adjust, evaluate again," your model becomes tuned to the quirks of that particular held-out set, just as it would have become tuned to the training data if you had evaluated there.

This is sometimes called leaking the evaluation set into the model-building process. It is the organisational equivalent of the law firm's mock jury — feedback from that jury gradually shapes how the lawyers argue, until the whole preparation is tailored to that specific group of people rather than to the broader task of persuading strangers.

The solution is a third pool of data: the test set. Think of it as the real jury. It is set aside at the very beginning and never looked at during model development. It is touched exactly once: at the end, when you have a finalised model and want a honest measurement of how it performs on genuinely new data.

The three roles, defined precisely

Each of the three splits has a distinct and non-interchangeable function. Mixing these functions is the central mistake that corrupts evaluation.

Training set: the precedent library

The training set is the data the model learns from directly. The optimisation algorithm — gradient descent in most modern ML systems — processes these examples, computes errors, and updates model parameters to reduce those errors. The model sees training data repeatedly, across many iterations called epochs.

Because the model actively adjusts itself to fit the training data, any performance measurement on training data tells you nothing trustworthy about generalisation. A model can achieve perfect accuracy on training data through memorisation. This number should never be used as the primary measure of model quality.

In the law-firm analogy, the training set is the library of known precedents. The lawyers read, re-read, analyse, and internalise them until they can argue from them fluently. But performing well in that reading room tells you nothing about how they will do in front of a real jury.

Validation set: the mock jury

The validation set is the data used to make modelling decisions. After each training run, or after each change to the model architecture or hyperparameters, you evaluate on the validation set to see whether the change helped. This is the feedback loop that drives model development.

Because you are actively using validation performance to guide decisions, the validation set is not a pristine, neutral evaluation. Information from it flows back into model-building choices. This is expected and deliberate — that is what the validation set is for. But it means validation performance is a slightly optimistic estimate of true generalisation performance, because the model has been indirectly shaped by it.

Typical modelling decisions driven by validation performance include: choosing between algorithm A and algorithm B; deciding how many layers a neural network should have; selecting a regularisation strength; determining when to stop training (a technique called early stopping, where you halt training when validation loss stops improving rather than letting the model continue to overfit the training data).

Test set: the real jury

The test set is the data that provides the final, honest measure of model performance. It is set aside at the beginning of the project and not examined until model development is complete. You do not tune hyperparameters based on test performance. You do not choose between models based on test performance. You do not look at test results and then go back and modify your model.

If you do any of these things, the test set is no longer a genuine held-out evaluation. It becomes another validation set, and you lose your one opportunity for an honest final measurement. This is why the rule is strict: the test set is touched exactly once.

In some organisations, the test set is stored in a separate location and only unlocked by a designated team member at deployment time. This is not paranoia — it is the procedural equivalent of the trial judge enforcing that the lawyers cannot speak to the real jury before proceedings begin.

Key Insight
The three-split discipline maps directly onto three distinct epistemic states. Training data is what the model can use to learn. Validation data is what you can use to make decisions. Test data is what no one touches until the very end. The moment any data crosses from one category to another, the corresponding guarantee disappears.

What data leakage looks like in practice

Data leakage is the general term for any situation where information from outside the legitimate training set contaminates the model-building process, making performance appear better than it will be on genuinely new data. It corrupts evaluation by breaking the clean separation between what the model knows and what is being used to measure it.

Leakage is distressingly easy to introduce without noticing. Here are four patterns that appear regularly in consulting and analytics projects.

Leakage through preprocessing

Suppose you are normalising a numerical feature — say, scaling revenue figures to a 0–1 range. If you compute the minimum and maximum values across the full dataset (including your test set) before splitting, you have let test-set information influence how training data is processed. The correct procedure is to compute the scaling parameters using training data only, then apply the same parameters to validation and test data.

The same issue arises with missing value imputation, outlier removal, and feature selection based on correlation statistics. Any statistic derived from the full dataset is carrying test-set information forward.

Leakage through target encoding

A common technique in categorical modelling is target encoding: replacing a category label with the average target value for that category. If you encode categories using statistics derived from the full dataset, test examples silently inform how training examples are represented. The fix is to encode categories using training data statistics only, applied consistently downstream.

Leakage through time

When your data has a time dimension, the split must respect chronological order. A model predicting tomorrow's sales must be trained on yesterday's data, not on a random sample that may include future values. Randomly shuffling a time-series dataset before splitting creates the fiction that the future is available as training material — and produces evaluation metrics that look excellent but are entirely fabricated.

The correct approach for time-series data is a temporal split: everything before date X goes into training, a window around X goes into validation, and everything after another cutoff goes into test. This mirrors real deployment conditions, where predictions are always about events that have not happened yet.

Leakage through the outcome variable

The most dramatic leakage occurs when the feature set includes information that is directly derived from the target variable, or that is only available because the outcome is already known. A churn model that includes "customer service calls about cancellation" as a feature may be inadvertently encoding the churn outcome itself, if those calls only happen right before a customer leaves. In deployment, you would not have that feature at prediction time.

This form of leakage often produces evaluation metrics that look impossibly good — which is precisely the signal to go looking for it.

Typical split ratios and the k-fold alternative

When you have ample data, a common starting division is roughly 70% training, 15% validation, and 15% test. The exact ratios matter less than maintaining the principle. With very large datasets — millions of examples — you may need only a few thousand examples for validation and test to get stable estimates, freeing more data for training. With small datasets, even 15% may not give you reliable statistics.

When to use k-fold cross-validation

With small datasets, a fixed split wastes precious training data. If you have only 500 labelled examples and hold out 150 for validation, you are training on 350 — a punishingly small sample for many problems. K-fold cross-validation solves this.

The idea is to partition training data into k equal folds (commonly 5 or 10). You run k training rounds. In each round, one fold serves as the validation set and the remaining k−1 folds serve as training data. At the end, you average performance across all k validation scores. This gives you a validation estimate that uses all available training data without contaminating any of it during each individual round.

The critical point: k-fold cross-validation still requires a separate, untouched test set. Cross-validation replaces the fixed validation split, not the test split. You still need a pool of data that was set aside before any decisions were made, against which you measure the final chosen model exactly once. Confusion on this point — treating k-fold results as the final evaluation — is a common mistake in project reporting.

Split before anything else

Before you explore your data, before you compute summary statistics, before you check for outliers: split off the test set and store it separately. Any analysis you run on the full dataset risks carrying test-set information into your preprocessing decisions.

Fit preprocessing to training data only

Compute all scaling parameters, imputation statistics, and encoding maps using training data alone. Apply the same computed parameters (without refitting) to validation and test data. This preserves the clean separation.

Iterate using validation performance

Build, train, and compare model variants using validation metrics. Use early stopping, hyperparameter search, and architecture choices guided by validation scores. This is the normal development loop.

Evaluate on test exactly once

When you have a finalised model, run it on the test set. Record the result. Do not use this number to make further decisions. It is your reported performance, and it represents how the model will behave on new data.

A note on the connection to overfitting

The three-split discipline does not prevent overfitting. It detects it. A model can still overfit the training data even with a perfectly maintained split. What the split gives you is the ability to notice: if training accuracy is 98% and validation accuracy is 67%, overfitting is happening and you can take corrective action — more regularisation, less model capacity, more training data, earlier stopping.

The approaches used to reduce overfitting are a separate topic. They include techniques like dropout in neural networks, regularisation in linear models, ensemble methods, and data augmentation. But none of them can substitute for a proper evaluation protocol. Without clean data splits, you cannot measure whether any of these techniques actually helped.

The consulting parallel: why this matters beyond ML

The intuition behind train/validation/test splits appears in any rigorous decision-making process. A management consultant preparing a market-entry recommendation uses historical case studies to develop hypotheses (training), stress-tests the recommendation against analogous scenarios the team brainstorms (validation), and ultimately faces client scrutiny in the final presentation — where the quality of reasoning is exposed without the safety net of internal review (test).

A diagnostic physician studies known presentations to build clinical pattern recognition (training), orders tests to refine differential diagnoses for the current patient (validation), and then commits to a treatment decision based on findings she cannot revise once treatment has started (test).

In both cases, the key principle is the same: the measure of true performance is always exposure to conditions that were not used to develop the approach. Internal rehearsal, however thorough, is not the same as external validity. The data-split discipline is the formal, machine-enforced version of this common-sense rule.

When exactly should you use k-fold cross-validation rather than a single fixed split? +

Use k-fold when your dataset is small (roughly fewer than a few thousand labelled examples) and a fixed validation split would leave too few examples for reliable performance estimates. K-fold uses all training data for validation across folds, giving a more stable estimate with less variance.

Keep a fixed single split when your dataset is large (thousands to millions of examples). At that scale, a fixed 15% validation split already contains enough examples for reliable statistics, and the computational cost of training k models is often impractical.

Use stratified k-fold when your target variable is imbalanced — for example, a fraud detection dataset where only 0.5% of transactions are fraudulent. Stratification ensures each fold contains the same proportion of minority-class examples, preventing folds where the rare class barely appears.

Always keep a separate test set either way. K-fold is a replacement for a fixed validation split, not for a held-out final test evaluation. Reporting k-fold cross-validation scores as your model's final performance is a common mistake that removes the guarantee of a truly unseen evaluation.

Key Insight

Every piece of data can only be in one role at a time. Training data teaches the model. Validation data informs the developer. Test data evaluates the final product honestly. The moment data crosses roles — through careless preprocessing, through peeking at test results, through random shuffles that ignore time — the evaluation is corrupted. Data leakage is not always deliberate; it is most dangerous when it is invisible.
