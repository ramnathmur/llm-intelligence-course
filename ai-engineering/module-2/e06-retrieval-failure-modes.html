Retrieval Failure Modes

The six ways RAG breaks in production and how to diagnose each one.

~15 min
Anchor Analogy
A search engine that returns ten results but the right answer is on page three. The problem might be indexing, ranking, or relevance — and each has a completely different fix. Blaming the last step (the answer it gives you) when the fault is in the first step (what it looked up) wastes days of debugging effort.
Why This Matters
Most RAG systems that fail in production are not failing because of LLM limitations — they are failing in the retrieval layer. Knowing the diagnostic taxonomy means you fix the right thing. Spending a week prompt-engineering your way around a vocabulary mismatch problem is the engineering equivalent of adjusting the mirror because the engine is misfiring.
After This Page, You Can Argue
How to distinguish retrieval failures from generation failures and which diagnostic to run first. You can walk into a post-mortem on a broken RAG system and correctly identify whether the problem is in indexing, chunking, ranking, vocabulary, context assembly, or the model itself — before anyone has run a single experiment.

The Diagnostic Problem

A RAG system is a pipeline with at least three stages: index the documents, retrieve relevant chunks, generate an answer. When the system gives a wrong or empty answer, the failure could have originated in any of these stages — or in the handoffs between them. The most common engineering mistake is to treat the symptom (a bad generation) as the disease (a generation problem) and to spend effort tuning the LLM when the root cause is twenty layers upstream.

Think of it like a hospital discharge summary that a consulting team relies on. If the summary was misfiled (indexing failure), never retrieved (retrieval failure), or truncated before the relevant paragraph (context assembly failure), no amount of coaching the consultant on how to read the summary will fix the problem. The fix lives in the document management system, not in the consultant's training.

This page gives you a structured taxonomy of the six most common retrieval failure modes. For each one, you will learn what it looks like from the outside, what is actually going wrong on the inside, and how to diagnose and fix it.

Key Insight
Always isolate the retrieval layer before touching the generation layer. Run your query against the index directly, inspect the top-K chunks, and ask: "If I handed these chunks to a human expert, could they answer the question correctly?" If yes, the problem is in generation. If no, the problem is in retrieval. This one heuristic eliminates most debugging dead-ends.

Failure Mode 1: Indexing Gaps

The first and most fundamental failure is the simplest: the answer is not in the index at all. The document was never ingested, the ingestion pipeline silently dropped it, or the document format was not supported and produced empty text.

This is the equivalent of asking a library for a book that is not in the catalogue. The librarian is not at fault for failing to find it.

What it looks like

The system confidently returns an answer that is not grounded in any retrieved document, or it says it cannot find the information — even though you know the document exists. When you inspect the retrieved chunks, none of them contain the relevant passage.

How to diagnose it

Run a direct keyword search or exact-phrase search against your document store using text you know appears in the missing document. If the document does not surface, the ingestion pipeline is the problem. Check ingestion logs for errors, verify the document was parsed correctly (especially PDFs with scanned images, which require OCR), and confirm the document is within any date or metadata filters your pipeline applies.

The fix

Audit your ingestion pipeline. Add a post-ingestion verification step that samples documents and confirms they are retrievable. For PDFs and scanned documents, add an OCR layer. For large document archives, run periodic index health checks that confirm document count and sample retrievability.

Failure Mode 2: Chunking Errors

Even when a document is correctly indexed, the way it was split into chunks can make the answer unretrievable. If the answer spans a chunk boundary — the first half is in chunk 47 and the second half is in chunk 48 — neither chunk alone will satisfy the query, and the retriever will rank both of them low because neither looks like a complete answer.

The analogy is a project report that has been photocopied and the staple was removed. The critical recommendation is split across two pages, but you only allowed your junior analyst to read one page at a time.

What it looks like

The retrieved chunks contain relevant vocabulary but miss the key sentence or paragraph. The answer is tantalizingly close in the index but never quite retrieved in one piece. The model hallucinates a plausible-sounding completion for the partial information it received.

How to diagnose it

Find the source document for the question you are testing. Locate the exact passage that answers the question. Then find which chunk or chunks that passage appears in. If it straddles a chunk boundary, or if it requires context from a preceding paragraph that is in a different chunk, you have a chunking problem.

The fix

Increase chunk overlap so that passages near boundaries appear in two adjacent chunks. Use semantically-aware chunking that splits on paragraph or section boundaries rather than fixed character counts (Shi et al., 2023). For structured documents (contracts, financial reports, technical manuals), use document-structure-aware parsers that preserve section context.

Failure Mode 3: Query-Document Vocabulary Mismatch

This is the most subtle and commonly misdiagnosed failure mode. The document uses one set of terms. The user's query uses different but semantically equivalent terms. The vector similarity between the query embedding and the document chunk embedding is low — not because the answer is absent, but because the language used to describe the same concept differs.

A medical professional asks "myocardial infarction treatment protocols." The document calls it "heart attack management guidelines." Dense retrieval systems trained on general text may encode these as similar, but sparse systems (BM25, keyword search) will fail completely because there is no lexical overlap.

What it looks like

The system fails on expert queries from domain specialists but works on casual queries from generalists. Or the reverse: specialist documents retrieved correctly when queried by experts but not by customers using everyday language. The failure is asymmetric and correlates with who wrote the query versus who wrote the document.

How to diagnose it

Take a failing query. Manually find the passage in the source documents that answers it. Extract the exact vocabulary used in that passage. Now compare that vocabulary to the vocabulary in the query. If the overlap is low but the meaning is the same, you have a vocabulary mismatch problem. Confirm by rewriting the query using the document's exact terminology and checking if retrieval succeeds.

The fix

Hybrid search addresses this directly. Combine a dense retrieval system (vector embeddings that capture semantic similarity) with a sparse retrieval system (BM25 or similar keyword-matching) and merge the results using a reranker or reciprocal rank fusion (Karpukhin et al., 2020). The dense system catches semantic equivalence; the sparse system catches exact matches. Together they cover the vocabulary gap from both sides.

An alternative is query expansion: use the LLM itself to generate multiple phrasings of the user's question before retrieval, then retrieve on all phrasings and deduplicate (Ma et al., 2023).

Self-Check
A user asks "how do I terminate my subscription?" and gets no useful answer. The word "terminate" never appears in any document — the documents all use "cancel". Which failure mode is this?

Failure Mode 4: Ranking Failures

Even when the right chunk is in the index and the query vocabulary matches, the retriever may rank irrelevant chunks higher. The correct chunk is retrieved — it is in the top-20 results — but it does not make the top-K cutoff that gets passed to the model. The answer exists, is findable, but is edged out by plausible-looking but wrong results.

Imagine your firm's knowledge management system returns ten results for a query, ranked by document recency. The correct answer is in a three-year-old policy document ranked ninth. The top eight results are recent but irrelevant. The junior analyst reads only the top three. The firm gets bad advice not because the answer does not exist, but because the ranking function chose the wrong criterion.

What it looks like

Manually inspecting the top-K retrieved chunks reveals that the correct chunk appears at position K+1 or K+2. The system works if you increase K, but this causes other problems (the model sees too much context, including irrelevant content). Performance is inconsistent: some queries retrieve correctly at K=5 while others need K=20.

How to diagnose it

For a set of known-answer test queries, retrieve the top-20 or top-50 results (much larger than your production K). Check at what rank the correct chunk appears. If the mean rank is consistently above your cutoff K, the retriever's scoring function is miscalibrated. Measure Recall@K: what fraction of your test queries have the correct chunk in the top K?

The fix

Add a reranking step after initial retrieval. Retrieve a larger candidate set (top-50 or top-100) using a fast first-stage retriever, then apply a slower but more accurate cross-encoder reranker to score each (query, chunk) pair jointly (Nogueira et al., 2019). Cross-encoders see the query and chunk together rather than comparing pre-computed embeddings, so they can make finer relevance judgments. This is the standard two-stage retrieval architecture in production systems.

Failure Mode 5: Context Window Overflow and Assembly Errors

Once the right chunks are retrieved and ranked, they must be assembled into the prompt sent to the model. Two things go wrong at this stage. First, if the total token count of the retrieved chunks exceeds the model's context window, some chunks get truncated or dropped. Second, the order in which chunks appear in the prompt matters: some models exhibit "lost-in-the-middle" behaviour, where they attend poorly to information in the middle of a long context (Liu et al., 2023).

What it looks like

The system works for short documents but degrades on long ones. Performance improves dramatically if you reduce the number of retrieved chunks. The model ignores information from chunks that appear in the middle of the assembled context, even when those chunks contain the correct answer.

How to diagnose it

Log the full assembled prompt for failing queries. Count the total tokens. Check whether any chunks were truncated. Then take the correct answer chunk and note its position in the assembled prompt. If it is consistently in the middle third, you may be encountering lost-in-the-middle effects. Test by reordering chunks to place the most relevant ones first and last.

The fix

Set a hard token budget for retrieved context and trim or summarise chunks to fit. Place the most relevant chunks at the beginning of the context (and if you have budget, at the end). Use a reranker score to determine which chunks are most relevant before assembling. For very long documents, use a summarisation step to compress low-ranked chunks before they enter the prompt.

Failure Mode 6: Query Understanding Failures

The final failure mode is in translating the user's intent into a retrieval query. This is distinct from vocabulary mismatch: here the problem is not the words but the structure of the query. A question that contains multiple sub-questions, implicit assumptions, or references to prior conversation context may be poorly served by a single retrieval call.

A user asks: "What was the board's rationale for the 2022 dividend change, and how does that connect to the current debt covenants?" This is two questions with an implied causal link. A single vector query will embed the whole sentence and likely retrieve documents about either dividends or covenants, but not the connective reasoning. The retriever cannot bridge two separate retrieval needs in one call.

What it looks like

The system handles simple factual queries well but degrades on complex, multi-part, or conversational queries. Performance drops when the user asks follow-up questions that reference earlier turns ("what about the other approach you mentioned?").

How to diagnose it

Decompose failing queries by hand into their constituent retrieval needs. If you identify two or more distinct retrieval needs that require separate document chunks, you have a query understanding failure. Check whether the system also fails on follow-up queries that require conversation context to resolve correctly.

The fix

Use query decomposition: have the LLM split complex queries into atomic sub-queries, retrieve independently for each, then synthesise the results before final generation (Patel et al., 2022). For conversational settings, add a query rewriting step that resolves coreferences by incorporating conversation history into the query before retrieval.

The Diagnostic Decision Tree

When a RAG system returns a wrong or empty answer, run this sequence before touching any model code:

Does the answer exist in any document?

Manually search the raw document store for the answer. If you cannot find it anywhere, the question is unanswerable from this corpus — this is a scope problem, not a system failure. If you find it, continue.

Is the answer in the index?

Run a direct retrieval query using the exact text from the relevant passage. If the chunk does not appear, you have an indexing gap or chunking error. Fix the ingestion pipeline and re-index. If the chunk appears, continue.

Is the correct chunk in the top-K results?

Retrieve the top-20 results for the original user query. Is the correct chunk in there? If not, you have a vocabulary mismatch or ranking failure. Add hybrid search or a reranker. If the chunk is in top-K, continue.

Is the correct chunk present in the assembled prompt?

Log the full prompt sent to the model. Confirm the correct chunk is present and not truncated. If it is absent or truncated, you have a context assembly failure. Reduce chunk count or token budget. If it is present, continue.

Can a human expert answer correctly from only the assembled prompt?

Show the assembled prompt (without the question) to a domain expert and ask them to answer the question. If they can and the model cannot, the problem is in generation. If they also cannot, the retrieval is still incomplete. Now you know where to focus.

Hybrid Search: The Most Important Architectural Fix

Across failure modes 3 and 4, the single most impactful architectural intervention is hybrid search. The term means combining dense retrieval (embedding similarity) with sparse retrieval (keyword matching) and merging the ranked lists.

Dense retrieval uses embedding models to encode both the query and each document chunk into a high-dimensional vector. Chunks are ranked by cosine similarity. This system excels at capturing semantic equivalence but can miss cases where exact terminology matters.

Sparse retrieval uses algorithms like BM25, which score each document based on how often the query terms appear and how rare those terms are in the overall corpus. This system excels at exact matches and domain-specific terminology but fails when the user's vocabulary differs from the document's.

Dense Retrieval (Embedding)
Sparse Retrieval (BM25)
  • Captures semantic equivalence ("terminate" = "cancel")
  • Works across paraphrases
  • Sensitive to embedding model quality
  • Can rank plausible-but-wrong chunks high
  • Exact terminology match (crucial for medical, legal, technical)
  • Fast and interpretable
  • Fails on vocabulary mismatch entirely
  • Does not understand synonyms or paraphrase

In production, hybrid search typically uses Reciprocal Rank Fusion (RRF) to combine the ranked lists from both systems. Each chunk receives a fused score based on its rank in each list, and the merged ranking is passed to a reranker for final ordering (Cormack et al., 2009). This two-stage pipeline — broad retrieval followed by precise reranking — is the current production standard for RAG systems that must perform reliably at scale.

Think Further

All six failure modes involve a different layer of the pipeline. But consider a situation where fixing one failure mode reveals another. You fix vocabulary mismatch with hybrid search, and now you can retrieve the right chunks — but the correct answer spans two chunks that are always split apart. Which failure mode surfaces next, and what does that tell you about the order in which to address them?

Generation Failures: How to Tell the Difference

A generation failure is when the retrieval is correct — the right chunks are in the prompt — but the model produces a wrong answer anyway. This happens for distinct reasons that require distinct fixes.

Common generation failures include: the model ignores the retrieved context and answers from its parametric memory instead (faithfulness failure); the model answers a slightly different question than the one asked (relevance failure); or the model correctly reports what the retrieved document says but the document itself was outdated (temporal failure).

The diagnostic test is the human expert test from step 5 of the decision tree above. If a human expert, given the assembled prompt, would answer correctly, the model is failing at generation. If the human expert also cannot answer, retrieval is still incomplete. This single test prevents the most common debugging mistake: tuning generation when retrieval is broken.

Connecting Back

The distinction between retrieval failures and generation failures maps directly onto the RAG architecture you studied in E04 — RAG Architecture. The indexing, retrieval, and generation components each fail independently. The diagnostic tree above is simply the RAG architecture pipeline read backwards from symptom to cause.

Key Insight

The most common production failure is not the most complex. Vocabulary mismatch and chunking errors together account for the majority of RAG performance gaps in real deployments. Before investing in rerankers, query decomposition, or larger context windows, confirm that your chunking strategy preserves complete passages and that your retrieval system uses hybrid search. Those two changes fix most systems.

A Note on Evaluation

You cannot diagnose retrieval failures systematically without an evaluation harness. Build a small golden dataset: twenty to fifty questions where you know the correct answer and the exact source passage. For each question, log the top-K retrieved chunks, note whether the correct chunk is present, and record the final model answer. Run this harness every time you change the pipeline.

Retrieval-specific metrics to track: Recall@K (is the correct chunk in the top K?), Mean Reciprocal Rank (what is the average rank of the correct chunk?), and Context Precision (of the chunks that were retrieved, what fraction were actually relevant?). These metrics attribute failures to the retrieval layer and give you the signal you need to iterate (Es et al., 2023).

Think first: What does it mean for hybrid search to address vocabulary mismatch but not chunking errors? +

Hybrid search merges dense (semantic) and sparse (keyword) retrieval. It improves which chunks surface at the top of the ranked list by handling cases where the query and document use different words for the same concept. But it cannot fix a case where the answer is split across two chunks, because neither chunk alone is semantically complete regardless of which retrieval method you use. Chunking errors need to be fixed at the indexing stage, not the retrieval stage. Hybrid search and chunking fixes are complementary, not substitutes: they live at different points in the pipeline.
