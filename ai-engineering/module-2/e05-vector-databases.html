Vector Databases

How semantic search works at scale — approximate nearest-neighbour search and when to use which store.

~15 min

Prerequisites: E04 — RAG Architecture

Imagine you have built a RAG system. Your knowledge base has 5 million documents, each converted to a 1,536-dimensional embedding vector. A user submits a query. You embed the query into the same space and want to return the 10 documents whose vectors are closest to the query vector. How long does that take?

If you compare the query vector against every document vector one by one — computing a cosine similarity for each — that is 5 million dot products. On a modern server, this takes roughly 2–5 seconds per query. For a real-time application, that is catastrophic. A user waiting 4 seconds for a search result will not wait long. This is the scaling wall that every production RAG system eventually hits, and it is exactly why vector databases exist.

This page covers how vector databases solve the problem, what they sacrifice to do it, and how to choose between them. By the end, you will be able to reason about retrieval failures not just as "the system returned the wrong document" but as a specific consequence of the index design choices made during setup.

The problem: exact search does not scale

The naive approach is called exact nearest-neighbour search or brute-force search. You compare the query to every vector in the database and rank them by similarity. The answer is always correct — you guaranteed the closest vectors — but the cost is linear in the number of vectors. Double the corpus, double the query latency.

Small corpora handle this fine. If you have 10,000 documents and queries run in 50 milliseconds, you are not worried. But the moment you cross the mid-hundreds-of-thousands mark, exact search becomes impractical for interactive applications. At millions of vectors and sub-second latency requirements, it is simply off the table.

The fundamental challenge is called the curse of dimensionality. In high-dimensional spaces (like the 1,536 dimensions of a typical embedding), the geometry behaves counterintuitively. Points that should be close in meaning end up at similar distances from each other, making it hard to prune large swaths of the search space early. Tree structures that work brilliantly in two or three dimensions degrade toward brute force as dimensionality climbs.

The solution is to give up the guarantee of exactness and instead find vectors that are approximately nearest. This is called Approximate Nearest Neighbour search, or ANN.

ANN: trading recall for speed

The ANN insight is deceptively simple: you do not need to check every book if you can quickly identify which section to check. Build an index structure at load time that organises vectors into navigable regions. At query time, use that structure to home in on the promising region and search only there.

This is a direct trade-off. You get fast queries. In exchange, you occasionally miss a true nearest neighbour because it sits in a region you did not visit. The miss rate is measured by recall — the fraction of the true top-K results that appear in your approximate top-K results. A system with 95% recall at K=10 means that, on average, you retrieve 9.5 of the 10 truly closest documents. The one you miss might be the most relevant document in your corpus. Or it might be indistinguishable from the ones you found. That depends on your data.

The tunable parameter that controls the trade-off is usually called ef (in HNSW) or nprobe (in IVF-based indices). Increase it and you visit more of the index, improving recall but increasing latency. Decrease it and you get faster queries at the cost of more misses. This dial is the first thing to reach for when a production RAG system shows retrieval failures.

How HNSW works

Hierarchical Navigable Small World (HNSW) is currently the dominant ANN algorithm for LLM retrieval applications. It was introduced by Malkov & Yashunin (2018) (Malkov & Yashunin, 2018) and has become the default in most vector databases, including Pinecone, Weaviate, Qdrant, and Chroma.

The name contains the whole architecture. To understand it, start with the concept of a navigable small world. Imagine a social network where every person knows a few close friends (near neighbours) and also a few people in completely different cities (long-range connections). This structure has the "small world" property: you can reach anyone in a small number of hops by first using the long-range connections to jump across the network, then using the short-range connections to close in precisely.

HNSW builds exactly this kind of graph for vectors. Each vector becomes a node. During index construction, each node is connected to a small number of neighbours. Some connections are short (to similar vectors) and some are longer (to more distant vectors that provide navigational shortcuts). The hierarchical part adds multiple layers. The top layer has only a few hundred nodes connected by long-range links. Lower layers are progressively denser, with shorter and shorter connections. The bottom layer contains all vectors.

A query enters the HNSW graph at the top layer, navigates via long-range links to the right region, then descends through progressively denser layers to find the approximate nearest neighbour (amber node) at the bottom.

When a query arrives, the search enters the top layer at a pre-selected entry point and greedily moves toward nodes that are closer to the query, using the long-range links to travel large distances quickly. When it finds a local minimum at the current layer, it drops down to the next layer and continues the greedy search from that point, now using shorter-range links for precision. By the time it reaches the bottom layer, it has already navigated most of the distance. It then explores a local neighbourhood of the bottom layer — controlled by the ef parameter — and returns the closest candidates found.

Why is HNSW dominant? Three reasons. First, query performance is outstanding — typically logarithmic in the corpus size, so doubling the corpus barely affects latency. Second, it achieves high recall with modest ef values. Third, it supports dynamic insertion — you can add new vectors to the index without rebuilding it from scratch. This last property matters enormously for real systems where new documents arrive continuously.

The IVF alternative

The other major family of ANN algorithms is Inverted File Index (IVF). The name comes from information retrieval, where an inverted index maps terms to document lists. Here the analogy is: cluster all vectors into groups (using k-means), then at query time search only the most relevant clusters.

Building an IVF index has two phases. First, train a clustering model on your vectors, producing a set of centroids — one per cluster. This is done offline. Second, assign every vector in the corpus to its nearest centroid. At query time, embed the query and find the nprobe nearest centroids, then search only the vectors assigned to those clusters.

IVF is fast at query time and very memory-efficient when combined with product quantization (PQ), which compresses each vector by representing it as a set of short codes. The trade-off is that building the index requires a training phase, and dynamic insertion is awkward — new vectors may not fit neatly into the existing clusters. IVF also requires you to choose the number of clusters at build time. For static, batch-loaded corpora, IVF with PQ (sometimes called IVF-PQ or IVFADC) is hard to beat on memory and throughput. For dynamic corpora with frequent updates, HNSW is usually the better choice.

What recall actually means in practice

When a vendor says their vector database achieves "95% recall at K=10," this is not an abstract quality score. It has direct consequences for your RAG pipeline. Here is how to think about it.

Suppose the user asks a question and the true best-matching document — the one that perfectly answers it — sits at position 1 in the exact search ranking. Your ANN index, with 95% recall at K=10, will include this document in its top-10 results 95% of the time. But 5% of the time, it will miss it entirely. Your LLM then has to answer the question without the best evidence. In a low-stakes application, this is a tolerable failure rate. In a medical information system or a legal research tool, it may not be.

Recall is not uniform across your corpus. It tends to be worst for vectors that sit in sparse or outlier regions of the embedding space — unusual documents, specialised terminology, content that does not cluster neatly with anything else. These are often the exact documents that matter most for hard or unusual queries. This is the retrieval failure mode to audit first: run your hardest test queries and check whether the misses are concentrated in particular document types or topics.

Think first: if you increase the ef parameter from 50 to 200 in HNSW, what changes and what stays the same? +

What changes: The search explores a larger candidate set at the bottom layer before returning results. This increases recall — you are less likely to miss the true nearest neighbour. Query latency also increases because more comparisons are made.

What stays the same: The index structure itself (the graph with its connections) does not change. The vectors stored are the same. The ef parameter only controls how many candidates are considered during search, not how the graph is built. Build-time parameters like M (the number of connections per node) are fixed at index creation.

The practical upshot: ef is a run-time dial you can adjust without rebuilding the index. This makes it the first parameter to tune when recall is too low.

Choosing a vector store

The market for vector databases has exploded since 2022. The choice is more about your operational context than raw algorithm performance, since most serious options implement HNSW or offer it as a default. Here are the relevant distinctions.

Managed cloud vs. self-hosted

Managed options (Pinecone, Zilliz Cloud, Weaviate Cloud) handle infrastructure, replication, and scaling for you. You pay per vector stored and per query. The cost model is predictable and the operational burden is low. The downside is data leaves your infrastructure, which matters for compliance-sensitive applications.

Self-hosted options (Qdrant, Weaviate OSS, Milvus, Chroma) run on your own servers or Kubernetes cluster. You control data residency, can tune hardware (GPUs significantly accelerate large ANN operations), and pay only for infrastructure. The operational overhead is real, especially at scale.

Embedded options (Chroma running in-process, FAISS as a library, LanceDB) are the right choice for prototyping and small-scale deployments. No external service to manage, sub-millisecond latency for moderate corpus sizes, zero network overhead. They break down when you need to serve multiple processes or scale horizontally.

Metadata filtering

A capability that separates production-grade vector stores from toy implementations is metadata filtering. In a real system, you rarely want to search all vectors. You want to search vectors belonging to a particular customer, dated within the last 30 days, or tagged with a specific document type. Without metadata filtering, you would have to retrieve a broad set and then discard irrelevant results at the application layer — wasting compute and degrading effective recall.

Implementations vary significantly. Some databases apply the metadata filter before the ANN search (pre-filtering), which guarantees you only search within the qualifying subset but requires the filter to leave enough vectors to search effectively. Others filter after the ANN search (post-filtering), which is faster but can return fewer than K results if many candidates are filtered out. Qdrant, Weaviate, and Pinecone have invested heavily in efficient pre-filtering; check the documentation for your chosen store carefully, as this is one of the trickiest implementation details in practice.

Hybrid sparse-dense retrieval

Dense vector search is powerful but has a specific weakness: keyword precision. If a user searches for a very specific product code, a person's unusual name, or a technical acronym, the embedding model may not have learned a useful representation for it. The vector for "SKU-47829-B" lands somewhere vague; every document about product codes looks similar. A lexical search (old-fashioned keyword matching) would nail this instantly because the exact string appears in the document.

Hybrid retrieval combines dense vector search with sparse keyword search (typically BM25, the standard information-retrieval ranking algorithm). Both retrieve a set of candidates. A fusion step — usually Reciprocal Rank Fusion (RRF) (Cormack et al., 2009) — combines the two ranked lists into a single result set. Weaviate and Qdrant support hybrid retrieval natively. Elasticsearch has added vector support specifically to enable this pattern.

When should you use hybrid? Use it whenever your corpus contains specific identifiers, proper nouns, technical codes, or terminology that embedding models are unlikely to generalise well. A general knowledge assistant can often get away with pure dense retrieval. A legal document system searching for specific case citations, or a product catalogue searched by part number, almost certainly needs hybrid retrieval.

Indexing vs. query latency: the build-time cost

ANN indices are not free to build. Constructing an HNSW index over a million vectors takes minutes to hours depending on the dimensionality and the M parameter (the number of connections per node). The larger M is, the higher recall you get at query time, but the longer the build and the more memory the index consumes. A typical starting point is M=16 or M=32. Going above M=64 rarely pays off unless you have very high recall requirements and abundant memory.

For IVF indices, the number of clusters (often called nlist) is the key build parameter. More clusters mean finer-grained partitioning and potentially higher recall — but they also require a larger nprobe at query time to cover enough of the space. A common rule of thumb is to set nlist to roughly the square root of the number of vectors, and nprobe to 5–20% of nlist.

An important operational reality: when you add new vectors to an HNSW index, they are inserted with all the right graph connections. But as your corpus grows and the data distribution shifts, the index can develop structural inefficiencies. Most production systems periodically rebuild the index from scratch to restore optimal performance. Budgeting for this rebuild time — and designing your system to serve queries from a previous index while the new one builds — is a real infrastructure concern.

The memory question

Vector search is fundamentally a memory-bound operation. A single float32 vector of 1,536 dimensions takes 6 KB. One million such vectors take 6 GB — before the index structure overhead. The HNSW graph itself adds 30–100% on top, depending on M. At 10 million vectors, you are looking at 60–120 GB of RAM just for the index. This is why many production deployments use GPU-accelerated search (FAISS with CUDA support can parallelize comparisons massively) or opt for quantization.

Product Quantization (PQ) compresses each vector by splitting it into sub-vectors and replacing each sub-vector with a short code drawn from a learned codebook. A 1,536-dimensional float32 vector (6 KB) can be compressed to a PQ code of 64–192 bytes — a 30x–90x reduction. The trade-off is a small but real degradation in recall because the compressed representations are approximate. For very large corpora where fitting everything in RAM is the binding constraint, PQ is indispensable.

Key Insight

Vector databases solve the scaling problem by trading recall for speed. HNSW organises vectors into a navigable multi-layer graph where queries descend from sparse long-range connections to dense short-range ones, reaching approximate nearest neighbours in logarithmic time. The ef parameter is the runtime dial that controls how much of the graph to explore — the first thing to reach for when retrieval quality degrades. For queries involving specific identifiers or technical codes, dense search alone will fail; hybrid sparse-dense retrieval is the correct architecture.
