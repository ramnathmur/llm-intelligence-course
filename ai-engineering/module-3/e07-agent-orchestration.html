Agent Orchestration Patterns

Tool calling, ReAct loops, stateful workflows, and the failure modes that arise at each layer.

~15 min
Anchor Analogy
A project manager coordinating subcontractors. A single subcontractor is predictable. A PM who delegates to five subcontractors whose work feeds each other faces cascading failures — a bad report from one poisons all downstream work. The PM's job is not just delegation; it is knowing where to insert checkpoints so a single bad delivery does not destroy the entire engagement.
Why This Matters
Single-turn LLM calls are predictable. Multi-step agents introduce compounding failure: one bad step cascades into subsequent bad steps. Architecture choices determine where the system can recover. Engineers who understand these patterns can build agents that fail gracefully and recover on cue, rather than silently producing wrong answers four steps downstream from the original error.

Prerequisites: E01 — Prompt Engineering as a System

An agent is an LLM that can take actions. That is the minimal definition. What separates a useful agent from a brittle one is the scaffolding around the model: the pattern that governs how the model decides to act, what state it maintains between steps, and where a human or a system check can intervene.

Three patterns dominate practical agent engineering today: tool calling, the ReAct loop, and stateful workflow orchestration. Each adds capability. Each also adds a new class of failure mode. This page builds a mental model for all three.

Layer one: tool calling

A base LLM generates text. A tool-calling LLM can also invoke functions. When the model decides it needs to search the web, run a SQL query, or send an email, it emits a structured request describing which function to call and with which arguments. Your application executes the function and returns the result to the model. The model then continues generating.

This is the foundation of all agent work. Without it, the model is a closed system — it can reason about the world but cannot touch it.

How tool calling actually works

The mechanics are simpler than they look. You define each tool as a JSON schema describing the function name, its parameters, and what each parameter means. You pass this schema to the model alongside your prompt. The model has been trained to recognise this schema format and to respond with a structured tool-call object when it decides it needs to invoke one (Anthropic, 2024).

The model does not execute the function. It only requests execution. Your application code receives the request, runs the function, and sends the result back in the next message. This separation between the model's decision and the action's execution is what allows you to add validation, logging, rate-limiting, and human approval at the seam.

Key Distinction

The model decides what to call and with which arguments. Your application code decides whether to actually execute it. That gap between decision and execution is where you install every safety control in your agent system.

Failure modes at the tool-calling layer

Tool calling introduces three distinct failure classes that do not exist in single-turn LLM use.

Wrong tool selection. The model picks a tool that is technically available but not appropriate for the current task. This happens when tool descriptions overlap in scope or are vague. A model given both a search_web tool and a search_internal_docs tool will occasionally use the wrong one if the descriptions do not make the boundary sharp. The fix is precise tool descriptions with explicit scope constraints and representative examples of when each tool applies.

Hallucinated arguments. The model calls the right tool but invents parameter values that do not exist in the data it has seen. A model calling a get_customer_record function may generate a plausible-looking customer ID that is not real. Validate all arguments before execution — never trust that the model's argument is well-formed just because the function name is correct.

Silent failure propagation. A tool call returns an error or an empty result. The model receives this result and continues reasoning as if the answer were simply "there is nothing there," rather than "the tool failed." Downstream steps then build on an absence of information that was actually a failure of retrieval. The fix is to return structured error objects rather than empty results, and to instruct the model explicitly on how to handle tool errors.

Layer two: the ReAct pattern

A single tool call is not an agent. An agent reasons, acts, observes the result, and then reasons again. The ReAct pattern — short for Reasoning and Acting — formalises this loop into an explicit structure that the model is prompted to follow (Yao et al., 2022).

In the ReAct pattern, the model is prompted to interleave two types of output. Thought steps are the model's internal reasoning: what it currently knows, what it is uncertain about, and what it intends to do next. Action steps are tool invocations. The model alternates: think, act, observe the result, think, act, until it reaches a conclusion.

Think Further

Before the ReAct paper, the dominant approach for tool-augmented reasoning was chain-of-thought alone: the model reasoned through the problem in text, then picked a tool. ReAct showed that interleaving reasoning with action — thinking between each tool call rather than only before the first one — dramatically improved accuracy on multi-step retrieval tasks. The intuition is that each new observation should update the reasoning, not just confirm a plan the model made at the start.

Why ReAct outperforms chain-of-thought alone

Pure chain-of-thought reasoning before tool use commits the model to a plan before it has seen any evidence from the environment. If the first tool call returns unexpected data, the model's subsequent reasoning is fighting against its own prior plan.

ReAct threads reasoning and observation together. After each tool result, the model can update its plan. This makes the agent adaptive rather than pre-planned. In the consulting analogy: a PM who updates the project plan each time a subcontractor delivers will catch bad work faster than one who reviews everything only at the end.

Empirically, ReAct outperformed chain-of-thought alone on HotpotQA and FEVER benchmarks by significant margins, particularly on questions that required pivoting mid-research when early assumptions proved incorrect (Yao et al., 2022).

Implementing the ReAct loop

The implementation is a loop in your application code, not a feature of the model API. The model generates a thought and an action. Your code parses the action, executes the tool, appends the result to the conversation as an observation, and calls the model again. This continues until the model generates a final answer rather than an action.

Three engineering concerns arise immediately.

Loop termination. The model might keep generating actions indefinitely if it cannot reach certainty. Set a hard maximum on the number of steps. Log every step. If the loop hits the maximum, surface the intermediate state to the caller rather than returning an empty error — partial reasoning is often more useful than a clean failure message.

Context window management. Each observation appended to the conversation consumes tokens. Long-running ReAct loops can overflow the context window. Compress older steps to summaries or use a rolling window that retains the most recent N steps plus the original question. Research on long-context performance suggests that models attend most reliably to the beginning and end of the context, so keeping the task specification at the top and the most recent observations at the bottom is the sensible default (Liu et al., 2023).

Parsing reliability. If you rely on the model formatting its output as Thought: ... Action: ... in natural text, a model update can silently break your parser. Use structured outputs or tool-calling APIs to elicit reasoning and actions in a schema-defined format rather than as free text. The schema is part of the contract; natural-language formatting is not.

The ReAct loop: each step is a model call. Your application code sits between each pair of boxes.

Layer three: stateful workflow orchestration

A free-running ReAct loop is powerful for open-ended tasks. It is unreliable for tasks that have required intermediate checkpoints, depend on external approvals, or must be resumed after failure. For those cases, you need stateful workflow orchestration.

The distinction matters. A ReAct loop holds all its state in the LLM context window. When the loop ends — by reaching the answer, hitting a token limit, or crashing — the state is gone. A stateful workflow externalises the state into a durable store. Every step's output is written to a database or message queue before the next step begins. If the system crashes between steps, it can resume from the last committed checkpoint rather than starting over.

The PM analogy made concrete

Return to the project manager coordinating subcontractors. An unstateful PM keeps the entire project plan in her head. When she takes sick leave, the project stalls. A stateful PM maintains a project management system: every task's status is recorded, every deliverable is logged, every dependency is explicit. A deputy can pick up exactly where she left off.

Stateful agent orchestration builds the equivalent of that project management system. Each step is a node in a directed graph. The edges represent dependencies. The state store records which nodes have completed and what they produced. The orchestrator reads the state, determines which nodes are now unblocked, and dispatches them.

Why stateful workflows are more reliable than free-running loops

Three properties of stateful systems directly address the failure modes of free-running loops.

Idempotent steps. If a step is externalised and logged, you can re-run it after a failure without re-running every step before it. In a free-running loop, re-running after a crash means restarting from the beginning, re-incurring all the latency and cost of completed steps, and risking re-running side effects (sending an email twice, deducting a payment twice). Stateful systems allow you to skip completed steps.

Explicit state transitions. In a free-running loop, the "state" is implicit in the LLM context window. When things go wrong, debugging requires re-reading the entire context to understand what the model believed at each step. In a stateful system, you can query the state store directly. "What did step 3 produce? What did the model decide to do next based on that?" is answerable without re-running anything.

Programmatic validation gates. Between steps in a stateful workflow, you can insert validation checks that are entirely outside the model. A step that generates a SQL query can be followed by a validator that parses the SQL syntax and checks that the referenced tables exist, before the query is ever executed. This is not something you can do cleanly in a free-running loop without breaking the loop's flow.

Key Insight

A free-running loop is fast to build and fragile at scale. A stateful workflow is slower to design and dramatically more reliable. The right choice depends on stakes: use a loop for exploratory, low-stakes tasks where failure cost is low. Use a stateful workflow for tasks that modify external systems, involve money, or must complete reliably over long time horizons.

Multi-agent systems and cascading failure

A single agent can be monitored. A system of agents whose outputs feed each other is a cascading failure machine if not carefully designed.

Consider a pipeline: Agent A gathers research, Agent B synthesises it, Agent C writes the final report. If Agent A returns partially incorrect research, Agent B synthesises it faithfully, and Agent C reports it accurately. The error from Agent A has now been amplified by two subsequent steps and appears in the final report wearing the authority of a fully processed document.

This is not a hypothetical. Research into LLM multi-agent frameworks found that error propagation through agent pipelines is a primary failure mode in production deployments, and that the errors are systematically harder to detect at the end of the pipeline than at the point of origin (Wang et al., 2024).

Four patterns that limit cascading failure

Confidence scoring at agent outputs. Each agent in the pipeline should express uncertainty alongside its outputs. If Agent A's research synthesis includes low-confidence sections, downstream agents should treat those sections differently than high-confidence ones. This requires designing output schemas that carry confidence metadata, not just the content itself.

Cross-agent validation. Rather than passing Agent A's output directly to Agent B, pass it to a validator agent first. The validator's only job is to check for internal consistency, factual plausibility, and completeness. This adds latency and cost, but it breaks the propagation chain at the first point where errors can still be caught cleanly.

Decomposition that limits dependencies. The deepest vulnerabilities in multi-agent systems are long dependency chains: A depends on B which depends on C which depends on D. An error at D cascades through the entire chain. Where possible, design parallel subgraphs: A and B both depend on D, but A and B do not depend on each other. Errors at D surface independently at two points, both of which are close to the source.

Human-in-the-loop checkpoints. Some decisions should not be delegated to an agent, regardless of how well the preceding steps went. Any action that is irreversible, high-cost, or high-visibility should require a human approval step. The engineering question is not whether to include human checkpoints but where to position them to maximise coverage while minimising interruption.

Human-in-the-loop: where and how

A human-in-the-loop checkpoint is not a fallback for when the model gets confused. It is a designed control point in the workflow, positioned based on risk analysis rather than ad hoc intuition.

Positioning checkpoints by risk profile

The right place for a human checkpoint is immediately before an action that is high-stakes and difficult to reverse. In the consulting analogy: you do not review every email the PM sends, but you do approve the final deliverable before it goes to the client and you approve any change to the project scope that incurs additional cost.

The engineering translation: define a risk tier for every action your agent can take. Tier 1 (read-only, reversible): search the web, read a file, query a database. No checkpoint required. Tier 2 (write, limited scope): create a draft, add a calendar event, send an internal message. Checkpoint optional based on confidence. Tier 3 (irreversible, external impact): send an email to a customer, submit a form, execute a financial transaction. Always checkpoint.

This tier structure can be enforced architecturally. Your tool-calling middleware checks the tier of each requested action and routes Tier 3 actions to a human approval queue before executing them. The model does not choose whether to ask for approval — the architecture enforces it.

Engineering a checkpoint that does not break the loop

The naive implementation of a human checkpoint pauses the agent and waits synchronously for a response. This is fine for short-lived tasks. For long-running workflows, you need an asynchronous implementation: the agent writes its checkpoint request to a queue, suspends its state, and resumes when the human response arrives — which might be hours or days later.

This requires the stateful workflow architecture described above. You cannot resume a suspended agent if all its state lives in an ephemeral context window. The checkpoint is only practical if the state is durable.

Active Debate
The research community has not converged on a standard for expressing agent uncertainty or confidence in multi-agent systems. Some frameworks attach numeric confidence scores to agent outputs; others use explicit reasoning traces; others rely on structured disagreement between multiple agents running in parallel. None of these has a clear empirical advantage across all task types. Treat any confidence mechanism you implement as a heuristic to be validated against your specific task distribution, not as a solved problem.
Self-Check
A multi-step agent is tasked with researching a competitor and drafting a strategy memo. The research step returns incomplete data. Which architectural choice best prevents this incomplete data from producing a confident-sounding but wrong strategy memo?

Putting the layers together

A production agent system typically combines all three layers. Tool calling gives the model the ability to act. The ReAct pattern gives the model a structured way to reason between actions. Stateful orchestration externalises the state so the system can recover, be audited, and be interrupted for human review.

Define tools with precise scope boundaries

Write tool descriptions that answer: when should the model use this, when should it not, and what does a well-formed argument look like? Overlapping tool scopes produce systematic misrouting. Validate all arguments before execution.

Implement the ReAct loop with a hard step limit

Prompt the model to produce explicit Thought/Action outputs. Parse using structured schema, not regex over free text. Set a maximum step count. Log every step to a durable store from the first loop iteration.

Externalise state before any long-running workflow

If the task runs longer than one synchronous response cycle, persists across sessions, or modifies external systems, move to a stateful orchestration architecture. Every step's output is committed before the next step begins.

Insert validation agents between producer and consumer agents

For any pipeline where Agent A's output feeds Agent B, add a validation step that checks completeness, confidence, and internal consistency. Fail loudly rather than passing bad data silently downstream.

Enforce human checkpoints architecturally for Tier 3 actions

Map every tool to a risk tier. Route Tier 3 actions (irreversible, external-impact) through a human approval queue. Do not make this a prompt instruction — make it a middleware enforcement that the model cannot route around.

Think Further

The three-layer architecture described here assumes a single orchestrator controlling a set of tools and sub-agents. Research into fully decentralised multi-agent systems, where agents negotiate tasks among themselves without a central coordinator, suggests different failure modes: deadlock, circular delegation, and goal divergence across agents with different context windows. These patterns are active research areas. Before adopting a decentralised architecture, ask whether the coordination overhead is justified by the parallelism gain, or whether a simpler centralised orchestrator would serve just as well with fewer failure modes to manage.
