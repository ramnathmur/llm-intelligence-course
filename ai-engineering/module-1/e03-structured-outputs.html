Structured Outputs

Getting models to return JSON, tables, and typed responses reliably in production systems.

~15 min
Anchor Analogy
Think of the difference between handing a consultant an expense form versus asking them to write up their expenses in an email. The narrative email is flexible and human-readable, but your accounting software cannot parse it. The form produces structured data directly — every field in its column, every number ready to sum. Structured outputs do for model responses what the expense form does for reimbursement requests: they impose a contract on the shape of the answer so downstream systems can process it without a human in the loop.
Why This Matters

Downstream systems that parse model output break when format varies. Structured outputs — using JSON schemas, tool-use APIs, or constrained decoding — make model responses machine-parseable and integration-safe. Without this, even a brilliant model answer is useless to any pipeline that needs to read a specific field, route on a value, or write to a database. Production AI engineering is largely the art of keeping data structures consistent across every hop in the chain.

Prerequisites: E01 — Prompt Engineering as a System

The Problem With Asking Nicely

When you write "respond only in valid JSON" in a system prompt, you are making a request. The model reads that instruction, weighs it against everything else in its context, and does its best to comply. Most of the time it succeeds. But "most of the time" is not a contract, and production systems need contracts.

The failure modes are subtle and varied. A model might return valid JSON but wrap it in a markdown code fence (```json ... ```), so your JSON.parse() call throws on the backticks. It might drop a trailing comma, or include a helpful explanatory sentence before the opening brace. It might faithfully produce JSON on ten thousand requests and then, on the ten-thousand-and-first, apologise for not being able to help and return plain English. Each of these events breaks a downstream consumer silently or loudly depending on how much error handling you built in.

The root cause is architectural: the model is a next-token predictor. When it is deciding what to generate, every possible continuation — valid JSON, malformed JSON, plain text, a refusal — is in competition. The system prompt nudges probabilities toward the valid-JSON region of the output space, but it cannot remove the other regions entirely. The model can always drift.

Why This Matters in Practice

In a proof-of-concept demo, you can add a try/catch, log the failure, and move on. In production you cannot absorb that kind of non-determinism. Consider a triage system that classifies incoming support tickets into urgency levels so they route to the right queue. If the model occasionally returns "high urgency" instead of "HIGH", tickets get lost. If it returns a paragraph explaining its reasoning instead of a classification label, the routing logic crashes. The cost is not just technical debt — it is real service-level failures for real customers.

This is why the engineering community has developed three distinct approaches to structured output, each offering a different trade-off between reliability, flexibility, and integration complexity.

Three Approaches to Structured Output

Approach 1 — Prompt Engineering (Soft Constraint)

You describe the format you want in the system prompt or few-shot examples. This is the oldest and most portable approach. It requires no special API features and works with any model that accepts text input.

The reliability ceiling is real but often acceptable for internal tools or low-stakes use cases. You can improve compliance by being explicit about edge cases ("if you have no value for a field, use null, never omit the key"), by providing a concrete example in the prompt, and by adding a post-processing step that validates the output against a schema and retries on failure. That retry loop is doing work the model should be doing, but it is pragmatic.

The deeper problem with prompt-only approaches is that the model has no way to know, as it generates each token, whether the string it is building is valid JSON. It is writing tokens sequentially, left to right, and only at the very end does the full response exist as a parseable object. It can miscount brackets, forget to close a string, or generate a key twice. These errors are not failures of instruction-following; they are failures of structure-awareness during generation (Liu et al., 2023).

Approach 2 — Tool Use / Function Calling (Structured API)

The major model providers now offer function-calling or tool-use APIs that separate the model's reasoning from the format of its output. Instead of asking the model to produce JSON, you define a function signature — a name, a description, and a schema for its parameters — and the API returns a structured function call object when the model decides to invoke it.

OpenAI's function calling (OpenAI, 2023), Anthropic's tool use, and Google's function declarations all work on this principle. The model produces a special token or structured block that the API layer interprets, rather than generating raw JSON as part of the text stream.

Reliability improves significantly for two reasons. First, the model has been fine-tuned on examples of correctly-formatted function calls, so it has a strong prior toward valid output. Second, and more importantly for some providers, the API layer applies lightweight validation: if the model produces a call that violates the schema (wrong type for an argument, missing required field), the provider can reject it and ask the model to retry internally, before the response reaches your code.

This approach is not the same as constrained decoding, but it sits closer to it than prompt engineering alone. The model still generates tokens freely, but it has learned to generate function call syntax reliably, and the infrastructure catches gross errors before they reach you.

Approach 3 — Constrained Decoding (Hard Constraint)

Constrained decoding changes how the model generates tokens, not just what it is asked to generate. At each step in the generation process, the set of valid next tokens is filtered according to a grammar or schema. If the model is currently inside a JSON string value, only characters that can legally appear inside a JSON string are allowed. If it is completing an integer field, only digit characters are valid. The model cannot produce malformed output because invalid tokens are masked out before the sampling step.

The most widely deployed implementation of this idea is Outlines (Willard & Louf, 2023), an open-source library that compiles a JSON schema (or a regular expression, or a context-free grammar) into a finite-state automaton. At each decoding step, the automaton tracks which state the partially-generated output is in and masks the logits of all tokens that would lead to an invalid transition. The model is effectively running inside a cage whose bars are defined by the schema.

This is a hard guarantee, not a probabilistic one. If you specify a schema that requires a field called priority with an enum of ["low", "medium", "high"], the model will always produce exactly one of those three strings for that field. It cannot produce "HIGH", "high urgency", or an explanatory sentence. The output is structurally valid by construction.

Important Nuance

Constrained decoding guarantees structural validity, not semantic correctness. The model might classify a low-urgency ticket as "high" — that is a reasoning error, not a format error. The cage ensures the key exists and the value is one of the three allowed strings. It does not ensure the model chose the right string. Format compliance and factual accuracy are orthogonal concerns.

JSON Schema as a Contract

Whether you use tool-use APIs or constrained decoding, you need to express what you want as a JSON schema. JSON Schema is a declarative vocabulary for describing the structure of JSON documents (Pezoa et al., 2016). It specifies field names, types, required vs. optional fields, enum constraints, string patterns, and numeric ranges.

Writing good schemas for AI output is slightly different from writing schemas for human-authored data. A few principles apply:

  • Use enums liberally. If a field can only take a small set of values, enumerate them. This is the clearest instruction you can give both the model and the constrained decoder. It eliminates capitalisation drift, synonym substitution, and creative paraphrase.
  • Mark fields as required or optional deliberately. Models tend to invent values for fields they cannot determine. If a field is optional, make it optional in the schema and instruct the model (in the prompt) to use null when the value is unknown. This is better than having the model hallucinate a plausible-sounding value.
  • Avoid deeply nested schemas when flat structures work. Deep nesting multiplies the number of ways the structure can be subtly wrong, and it increases the length of the generated output, which increases the chance of error accumulation.
  • Use additionalProperties: false. This prevents the model from adding extra fields beyond those you defined. Extra fields are often harmless, but they can cause downstream schema validators to reject the object.

The Streaming Conflict

Streaming is the technique of sending model output token-by-token to the client as it is generated, rather than waiting for the full response. From a user-experience standpoint, streaming is almost always better: the user sees text appearing immediately rather than staring at a blank screen for several seconds. It also reduces perceived latency for long responses.

Structured outputs and streaming create an architectural tension. A JSON object is not valid until the closing brace arrives. If you are streaming the response and trying to parse it as JSON, you will fail on every intermediate chunk because the object is not yet complete. You have several options:

Buffer and parse at the end. Collect all streamed chunks, concatenate them, and parse the result as JSON once the stream closes. This gives you all the reliability benefits of structured output but loses the responsiveness benefit of streaming. For internal API calls where the user sees no UI, this is usually the right choice.
Stream and parse incrementally. Use a streaming JSON parser that can handle partial input and emit events as fields are completed. Libraries like jsonstream (Node.js) or ijson (Python) do this. This lets you start acting on completed sub-objects before the full response arrives. The trade-off is added complexity: you must handle the case where a streaming parse fails midway.
Use a structured-first API with streaming disabled. If you are using a provider's tool-use API, check whether it supports streaming structured calls. Some providers buffer the function call arguments internally and only send them as a complete block even when streaming is enabled. Read the provider documentation carefully; behaviour varies.
Separate the streaming and structured concerns. Generate a streaming conversational response for the user, then make a second, non-streaming API call to extract structured data from the conversation. This doubles the API calls but cleanly separates user experience from data extraction. It works well when the structured data is a side effect of the interaction rather than the primary output.
Key Insight

The choice between streaming and structured output is not binary. Architect your system so that streaming serves the human-facing response and a separate extraction step produces the machine-parseable record. This pattern appears throughout production AI systems: the model speaks to users in natural language and simultaneously writes to databases in structured form.

Retry Logic and Validation Layers

Even with tool-use APIs, you should build a validation layer. Parse the returned structure against your schema using a library (jsonschema in Python, zod in TypeScript), and if validation fails, capture the error message and retry with a corrected prompt. Include the failed output and the specific validation error in the retry prompt so the model knows exactly what went wrong.

A common pattern looks like this:

def extract_structured(prompt: str, schema: dict, max_retries: int = 3) -> dict:
    for attempt in range(max_retries):
        response = call_model(prompt, schema=schema)
        try:
            validate(instance=response, schema=schema)
            return response
        except ValidationError as e:
            prompt = repair_prompt(prompt, response, str(e))
    raise StructuredOutputError("Failed after max retries")

The repair_prompt function appends a correction instruction: "Your previous response failed schema validation with the following error: [error message]. Please correct only the structural issue and return the same data in valid format." This is more effective than starting over because it preserves the reasoning the model already did and focuses it on the formatting error.

In practice, with a good model and a well-designed schema, you will see validation failures on fewer than 1% of calls. The retry adds a small latency cost but keeps your pipeline robust against that tail of edge cases.

Choosing the Right Approach for Your Use Case

The three approaches form a spectrum from flexible but fragile to rigid but reliable. The right choice depends on what you are building:

Use prompt engineering when…
Use constrained decoding / tool APIs when…
  • You are prototyping and iteration speed matters more than reliability
  • The output structure is complex and deeply nested, making schema definition cumbersome
  • You are using a model or provider that does not support function calling
  • The format is semi-structured (markdown with specific sections rather than strict JSON)
  • A downstream system parses the output programmatically with no human review
  • Format failures cause service disruptions or data loss
  • You need enum-level guarantees on specific fields (classification labels, status codes)
  • You are in production and latency from retry loops is unacceptable

A Worked Example: Support Ticket Classification

Suppose you are building a system that reads incoming support tickets and classifies each one before routing it. You need three fields: a priority level, a category, and a confidence score.

A prompt-engineering approach might include: "Respond with a JSON object with keys: priority (one of low, medium, high), category (one of billing, technical, account, other), and confidence (a number between 0 and 1)." This works most of the time.

A tool-use approach defines the schema formally and passes it to the API:

{
  "name": "classify_ticket",
  "description": "Classify a support ticket by priority and category",
  "parameters": {
    "type": "object",
    "properties": {
      "priority": {
        "type": "string",
        "enum": ["low", "medium", "high"]
      },
      "category": {
        "type": "string",
        "enum": ["billing", "technical", "account", "other"]
      },
      "confidence": {
        "type": "number",
        "minimum": 0,
        "maximum": 1
      }
    },
    "required": ["priority", "category", "confidence"],
    "additionalProperties": false
  }
}

Now the model cannot produce "HIGH", cannot add an explanatory field, cannot return a confidence of 1.5. The contract is enforced at the infrastructure level. Your routing logic can trust the response without defensive coding.

Self-Check
A model is asked to produce a JSON object with a status field that must be one of ["open", "closed", "pending"]. You use constrained decoding with the correct schema. On the next request the model returns "closed" for a ticket that should be "open". What type of error is this?
Think first: why does a system prompt instruction to "return only valid JSON" sometimes produce a response wrapped in markdown code fences? +

Because the model is predicting tokens, not following rules. During pre-training and fine-tuning, the model saw enormous amounts of code shared in chat-like contexts, and in those contexts JSON was almost always inside a markdown fence. When you ask for JSON, the model's learned associations pull it toward the most common pattern it has seen for presenting JSON in a conversational setting — which is the fenced block. Your instruction says "valid JSON" but the model's statistical priors say "JSON goes in a code block." The instruction shifts the probability distribution; it does not rewrite it. Constrained decoding or a tool-use API removes the competition entirely: the model simply cannot produce backtick tokens in that context.

Schema Design Pitfalls

Even with the right infrastructure, poorly designed schemas cause problems. Three pitfalls come up repeatedly in production systems:

Pitfall 1: Overloading a single field

Resist the temptation to put complex information into a single string field with instructions like "summarise the issue in one sentence, then list the steps taken." The model will follow this instruction inconsistently. Break composite fields into separate fields. Give the model one job per field and it will do each job more reliably.

Pitfall 2: Optional fields that get hallucinated

If you have an optional field for, say, a customer account number that can only be extracted if the ticket mentions it explicitly, the model will hallucinate plausible account numbers when none is mentioned. Mitigate this by making the field nullable ("type": ["string", "null"]), adding a second boolean field called account_number_found that the model sets to true only when it is confident, and instructing the model explicitly to return null when uncertain rather than guessing.

Pitfall 3: Schemas that are too permissive

A schema that accepts any string for a field is almost no constraint at all for a model that has already been asked to produce JSON. The value of a strict schema comes from its constraints: enums, numeric ranges, required fields, no additional properties. The more constraints you can express in the schema itself, the less you depend on the model's instruction-following, and the more reliable your pipeline becomes.

Think Further

As LLMs become more capable at following complex instructions, does the distinction between prompt-based soft constraints and hard-enforced structured outputs become less important? Or does the reliability gap between "model tries to follow a rule" and "rule is architecturally enforced" remain meaningful regardless of how good the model gets? Consider what happens at the tail of the distribution — the rare, edge-case input that breaks your system at 3 AM on a Saturday.

Key Insight

Prompt instructions and structural enforcement are not the same thing. A system prompt nudges the model's probability distribution toward the format you want. Constrained decoding or a tool-use API removes invalid tokens from consideration entirely. The difference is the difference between asking a contractor to use a specific form and handing them a form that only has boxes for the fields you need — one relies on compliance, the other on architecture.
