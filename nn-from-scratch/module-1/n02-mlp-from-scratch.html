Picture a new analyst at a consulting firm. Their manager hands them a financial model and says "update the revenue projection." The analyst opens the spreadsheet, sees rows of formulas, and — crucially — has no idea what any cell is doing. They cannot trust the output because they do not understand the machine beneath it.

Now picture the same analyst who built that model from scratch in a blank spreadsheet: typed each formula, linked each cell, debugged each circular reference. They can read the output. They know exactly which assumption drives which number. They can catch errors the first analyst would never see.

That is the point of this page. We are going to build a multi-layer perceptron (MLP) from blank NumPy arrays — no PyTorch, no Keras, no model.fit(). When we are done, you will know what a "layer" is because you wrote one. You will know what a "loss" is because you computed it. You will know what "training" is because you did it, line by line.

After that, every framework becomes a shortcut you understand, not a magic wand you trust blindly.

What an MLP actually is

An MLP is the simplest neural network worth calling a network. It is a chain of linear transformations with non-linear activations sandwiched between them. Each stage takes a vector of numbers, mixes them together into a new vector, then squashes each number through a non-linearity. Stack enough of these stages and the network can learn to approximate almost any function — a result known as the Universal Approximation Theorem (Hornik et al., 1989).

In business terms: imagine a chain of department heads, each of whom takes a report from the previous department, re-weights the information according to what they care about, then passes a new summary to the next department. The CEO at the end makes a decision. Training the network means adjusting what each department head cares about — their weights — until the CEO consistently makes the right call.

The three components

Every MLP has exactly three kinds of things:

1. Layers. Each layer is a weight matrix W and a bias vector b. Given input x, a layer computes W @ x + b. That is a matrix multiply followed by an addition — a linear transformation. Nothing more mysterious than a spreadsheet SUMPRODUCT.

2. Activations. After each linear layer, we apply a non-linear function element-wise. Without activations, stacking ten linear layers is exactly the same as having one — because a product of linear transformations is still linear. The activation is what makes depth useful. The simplest modern activation is ReLU: max(0, x). Every negative number becomes zero. Every positive number passes through unchanged. That is the whole thing.

3. A loss function. The loss measures how wrong the network's output is. For a classification task, cross-entropy loss is standard (Bishop, 1995). For a regression task, mean squared error works. The loss is a single number. Training means making that number smaller.

Building the forward pass

Let us build a network that classifies a toy problem: given a 2D point, predict which of two classes it belongs to. The network will have one hidden layer with 4 neurons.

Start with the data. We will work with a mini-batch of 8 examples at a time. Each example has 2 features (the x and y coordinates of the point).

Represent data as a matrix

Stack the 8 examples into a matrix X with shape (8, 2). Each row is one example. Labels go into a vector y of shape (8,) with values 0 or 1. This is the first design choice: process a batch of examples together, not one at a time.

import numpy as np

np.random.seed(42)
X = np.random.randn(8, 2)   # shape (8, 2)
y = (X[:, 0] + X[:, 1] > 0).astype(int)  # label: 1 if above diagonal

Initialise the weight matrices

Layer 1 maps from 2 inputs to 4 hidden neurons. Its weight matrix W1 has shape (2, 4) and bias b1 has shape (4,). Layer 2 maps from 4 hidden neurons to 2 output logits. Its weight matrix W2 has shape (4, 2).

We scale by 0.01 to keep the initial values small. You will see exactly why this matters below.

W1 = np.random.randn(2, 4) * 0.01
b1 = np.zeros(4)
W2 = np.random.randn(4, 2) * 0.01
b2 = np.zeros(2)

Run the forward pass

Matrix multiply, add bias, apply ReLU, repeat, then compute logits. All four operations in 4 lines.

# Layer 1
z1 = X @ W1 + b1       # shape (8, 4)
a1 = np.maximum(0, z1) # ReLU activation

# Layer 2 (output logits)
z2 = a1 @ W2 + b2      # shape (8, 2)

Compute the loss

Convert logits to probabilities with softmax, then measure cross-entropy loss against the true labels.

# Softmax: stable version (subtract max for numerical safety)
e = np.exp(z2 - z2.max(axis=1, keepdims=True))
probs = e / e.sum(axis=1, keepdims=True)  # shape (8, 2)

# Cross-entropy loss (mean over the batch)
correct_logprobs = -np.log(probs[np.arange(8), y])
loss = correct_logprobs.mean()

That is the complete forward pass. It is 10 lines of NumPy. The network has gone from raw inputs to a single loss number. Nothing hidden, nothing automatic. Every intermediate tensor has a name and a shape you can inspect.

Key Insight

A neural network forward pass is a sequence of matrix multiplications and element-wise functions. Each operation has an explicit input shape, output shape, and parameterization. No operation is mysterious. "Deep learning" means chaining more of these — the depth of the chain is what changes, not the kind of operation.

The backward pass: propagating gradients by hand

On the previous page, N01 — Autograd, you saw how gradients flow backwards through a computation graph. Now we apply that understanding to the MLP. We need the gradient of the loss with respect to every weight and bias so we can adjust them.

Think of it like a post-project debrief. The final outcome (loss) was bad. We work backwards: which department's output made the final decision go wrong? And which inputs into that department caused that bad output? We trace accountability back to the root decisions — the weights.

Gradient of the loss with respect to the output logits

For cross-entropy loss with softmax, the gradient with respect to z2 (the output logits) has a beautifully clean form. For each example, the gradient is: the softmax probability minus 1 for the correct class, and the softmax probability unchanged for every other class.

# Backward pass
dz2 = probs.copy()
dz2[np.arange(8), y] -= 1
dz2 /= 8  # divide by batch size (derivative of the mean)

If the correct class had probability 0.9, the gradient is −0.1 (you were close, small adjustment). If it had probability 0.1, the gradient is −0.9 (you were badly wrong, large adjustment). The math automatically scales the update by how wrong we were.

Gradients for the second layer

The second layer computed z2 = a1 @ W2 + b2. By the chain rule (Rumelhart et al., 1986):

dW2 = a1.T @ dz2         # shape (4, 2)
db2 = dz2.sum(axis=0)    # shape (2,)
da1 = dz2 @ W2.T         # shape (8, 4)

Notice the transpose pattern. When a forward pass does a1 @ W2, the backward pass does dz2 @ W2.T. This is always the case for matrix multiplication: the gradient flows through the transpose. It is the same rule every time — you just need to keep the shapes consistent.

Gradient through the ReLU

ReLU set negative values to zero in the forward pass. In the backward pass, it simply blocks the gradient for those positions: wherever z1 < 0, the gradient is zero; elsewhere, it passes through unchanged.

dz1 = da1 * (z1 > 0)    # ReLU backward: gate the gradient

Gradients for the first layer

dW1 = X.T @ dz1          # shape (2, 4)
db1 = dz1.sum(axis=0)    # shape (4,)

That is the entire backward pass. 6 lines. We now have gradients for all four parameter arrays: dW1, db1, dW2, db2. Each gradient has the same shape as its corresponding parameter, which is always true by definition.

The parameter update: gradient descent

Now we use the gradients to update the weights. The update rule for vanilla gradient descent with learning rate lr is:

lr = 0.1

W1 -= lr * dW1
b1 -= lr * db1
W2 -= lr * dW2
b2 -= lr * db2

We are moving each parameter in the direction that reduces the loss. The learning rate controls how big a step we take. Too large and we overshoot. Too small and training crawls. Choosing a learning rate is one of the core practical decisions in deep learning — a subject we return to in later modules.

Wrap this in a loop and you have a training run:

for epoch in range(1000):
    # Forward pass
    z1 = X @ W1 + b1
    a1 = np.maximum(0, z1)
    z2 = a1 @ W2 + b2

    # Loss
    e = np.exp(z2 - z2.max(axis=1, keepdims=True))
    probs = e / e.sum(axis=1, keepdims=True)
    correct_logprobs = -np.log(probs[np.arange(8), y])
    loss = correct_logprobs.mean()

    # Backward pass
    dz2 = probs.copy(); dz2[np.arange(8), y] -= 1; dz2 /= 8
    dW2 = a1.T @ dz2; db2 = dz2.sum(0)
    da1 = dz2 @ W2.T
    dz1 = da1 * (z1 > 0)
    dW1 = X.T @ dz1; db1 = dz1.sum(0)

    # Update
    W1 -= lr * dW1; b1 -= lr * db1
    W2 -= lr * dW2; b2 -= lr * db2

    if epoch % 100 == 0:
        print(f"Epoch {epoch}, Loss: {loss:.4f}")

The loss falls with each epoch. That is the entire learning algorithm.

Self-Check: Mini-batches
Why does training on a mini-batch of 8 examples produce better gradient estimates than training on a single example at a time?

Now: the three questions you must be able to answer

Three concepts underpin everything you just built. Each one has a non-obvious reason that frameworks hide from you.

Why small random weights, not zeros?

Initialise all weights to zero and every neuron in a layer computes exactly the same value. Every neuron has the same gradient. Every neuron gets the same update. After a thousand epochs, they are still identical — the layer has collapsed to a single neuron pretending to be four. This is the symmetry problem: neurons with identical weights can never specialise, because they receive identical gradients and make identical updates forever (Glorot & Bengio, 2010).

Small random values break symmetry. Each neuron starts at a different position in weight space. Each computes a slightly different value. Each receives a slightly different gradient. They diverge and specialise. The "small" part matters too: large random values produce large activations, which can saturate activation functions or cause exploding gradients. We scale by 0.01 as a safe starting point.

Think first: if two neurons start with identical weights, what happens after the first update? +

Both neurons compute the same output. The loss is the same regardless of which neuron produced each contribution. The gradient for both is identical. The update is identical. After the first update, they are still identical — just shifted by the same amount. The symmetry is never broken. The network behaves as if it only has one neuron in that layer for the entire training run. Zero initialisation wastes all but one neuron in every layer.

Why mini-batches produce better gradient estimates

A single training example is like reading one customer review before deciding whether to stock a product. The review might be atypical. The customer might have been having a bad day. A decision based on one data point is noisy.

A gradient computed from a single example is the same thing: it points in the right direction on average, but has high variance. One outlier example can send the update in a misleading direction.

A mini-batch averages the gradient over multiple examples. Individual noise cancels. The estimate of the "true" gradient direction — the gradient you would get if you could use the entire dataset at once — becomes more reliable. In practice, mini-batches of 32 to 256 examples balance noise reduction against computational efficiency. The code above uses 8 to keep the arithmetic visible.

There is a second, subtler benefit: the noise in stochastic gradient descent actually helps escape shallow local minima. Pure full-batch gradient descent can get trapped; the randomness of smaller batches keeps the optimiser exploring. This is one reason deep networks often train better with mini-batches than with the full dataset at once (Keskar et al., 2017).

Why vectorisation beats a Python for-loop

Imagine computing the gradient update for each of the 8 examples with a Python for loop: compute the forward pass for example 1, then example 2, and so on. That loop calls Python bytecode for every iteration. Python is interpreted; each loop step carries overhead.

NumPy's matrix multiply (@) sends the computation to a highly optimised BLAS library written in C. All 8 examples are processed in a single call, using SIMD instructions that operate on multiple numbers simultaneously. On modern hardware, the matrix multiply is typically 10–100x faster than the equivalent Python loop — and the gap widens to 1000x or more when you move to a GPU (Nvidia, 2020).

This is why machine learning practitioners speak about vectorisation almost reverentially. It is not a micro-optimisation. It is the difference between a training run that takes an hour and one that takes a week. Every framework operation you call in PyTorch or JAX is, under the hood, a vectorised kernel. When you write a Python loop where a matrix operation would do, you forfeit that entire acceleration.

For-loop (slow)

For each of N examples: compute z1, a1, z2 separately. Python overhead per step. Cannot parallelise across examples. On a GPU, you are wasting 99% of the hardware.

for i in range(N):
    z1_i = X[i] @ W1 + b1
    a1_i = np.maximum(0, z1_i)
    z2_i = a1_i @ W2 + b2
Vectorised (fast)

All N examples in one call. NumPy dispatches to BLAS. On a GPU, thousands of matrix cells update in parallel. Same mathematical result, 10–1000x faster.

z1 = X @ W1 + b1       # (N, 4)
a1 = np.maximum(0, z1) # (N, 4)
z2 = a1 @ W2 + b2      # (N, 2)

What you have actually built

Let us take stock. In around 30 lines of Python, you built:

A forward pass that maps raw inputs through two learned transformations to a probability distribution over classes. Every intermediate value — z1, a1, z2, probs — has a name and a shape. Nothing is invisible.

A loss function that converts the network's output into a single number measuring how wrong it is. The number is differentiable with respect to every parameter — which is the entire reason we chose these functions rather than, say, accuracy.

A backward pass that computes gradients using the chain rule. No automatic differentiation engine — you applied the calculus yourself, which means you can debug any gradient that goes to zero or infinity instead of staring at a framework error message.

A training loop that iterates forward, backward, and update until the loss falls. That loop is the same loop PyTorch's model.fit() runs internally — except now it is not hidden from you.

Everything a modern deep learning framework does is an acceleration and an abstraction of this loop. Automatic differentiation (autograd) automates the backward pass. GPU kernels accelerate the matrix multiplies. The optimizer object (SGD, Adam) replaces the weight update line with something more sophisticated. But none of those additions changes the fundamental structure: forward, loss, backward, update, repeat.

Think Further

This MLP has one hidden layer. What changes if you add a second? Map out: what new weight matrices are needed, what new lines appear in the forward pass, what new gradient calculations appear in the backward pass. Try writing it out before looking at a framework implementation.
