Autograd — How Gradients Flow

Every modern framework secretly builds a computational graph and walks it backward

~15 min
Anchor Analogy
Think of an audit trail. Every calculation in a forward pass is logged in a ledger. The backward pass traces the ledger in reverse, computing how each decision contributed to the final error — exactly like a financial audit that works backward from a discrepancy to the original entry that caused it.
Why This Matters

PyTorch, TensorFlow, and JAX all work the same way under the hood: they build a graph of operations and differentiate it automatically. When your training loop behaves unexpectedly — gradients vanish, explode, or simply do nothing — the explanation almost always lives in the computational graph. Understanding autograd makes framework behaviour legible. You stop treating the framework as a black box and start reading what it is actually doing.

After This Page, You Can Argue

How automatic differentiation works, why computational graphs must be stored during the forward pass, why backpropagation requires the chain rule rather than simple division, and what zeroing the gradients before each update actually does and why skipping it silently corrupts training.

Prerequisites: None — this is the entry point.

The Problem That Autograd Solves

Training a neural network is an optimization problem. You have a function — the network — that turns inputs into predictions. You have a loss function that measures how wrong those predictions are. Your goal is to adjust the network's parameters so the loss falls.

The adjustment rule is simple in principle: move each parameter in the direction that reduces the loss. To know which direction that is, you need the gradient of the loss with respect to every parameter. For a network with millions of parameters, computing those gradients by hand is not an option.

This is what automatic differentiation, or autograd, solves. It is a system that takes any computation you define in Python and computes its exact gradient automatically. PyTorch calls it autograd. TensorFlow calls it GradientTape. JAX calls it grad. The mechanism is different in each case, but the idea is identical.

Why Not Just Estimate the Gradient?

You might wonder: why not estimate the gradient numerically? Take the loss, nudge each parameter slightly, measure how the loss changes, divide. This is called finite differences, and it works in theory. In practice, a network with ten million parameters requires twenty million forward passes to estimate all gradients. That is prohibitively slow. Autograd computes all gradients in a single backward pass — a feat made possible by the chain rule of calculus.

The Computational Graph

When you run a forward pass in PyTorch, Python is not just computing numbers. It is building a record of every operation performed. This record is the computational graph.

Imagine you write:

import torch

x = torch.tensor(3.0, requires_grad=True)
y = x ** 2       # y = x^2
z = y * 2 + 1   # z = 2x^2 + 1

Behind the scenes, PyTorch creates three nodes: one for x, one for y, one for z. It also records the edges between them — the operations that connect them. The ** operator created a squaring node. The * and + operators created multiplication and addition nodes. Each node stores a reference to the operation that produced it and a reference to its inputs.

This structure is a directed acyclic graph (DAG). Directed because information flows one way — from inputs toward outputs. Acyclic because no value depends on itself.

Key Insight

The computational graph is not an optimization choice. It is a necessity. To compute the gradient of the final loss with respect to an early parameter, you must know every operation that sits between them. The graph is that record. Without it, backpropagation has nothing to trace.

What Gets Stored and What Does Not

PyTorch only tracks operations involving tensors with requires_grad=True. Your input data — images, text tokens, numbers — typically does not require gradients. Only the model's parameters do. This distinction matters: it controls which parts of the graph get built, and therefore which parts participate in the backward pass.

By default, PyTorch builds the graph as a dynamic structure: it is rebuilt fresh on every forward pass. This is why PyTorch's autograd is called "define-by-run." TensorFlow 1.x used a static graph, defined once and compiled before any data was run through it. JAX uses a different model again, based on function transformations. The dynamic approach is more flexible — you can write conditionals and loops in ordinary Python and they will be captured in the graph automatically.

The Forward Pass: Building the Ledger

The forward pass does two things at once. First, it computes the output values — this is what you see. Second, it records the operations used to compute them — this happens invisibly.

Each operation node in the graph stores a backward function: a small piece of code that knows how to compute the local gradient of that operation given the upstream gradient. The **2 operation stores the rule for differentiating a square: bring down the exponent, reduce the power by one. The * operation stores the rule for differentiating a product.

Think of the audit trail analogy. During the financial year, every transaction is recorded in a ledger: who paid whom, what the amount was, what account was affected. The recording happens as a byproduct of running the business. At audit time, the auditor does not need to re-run the business — the ledger is already there.

In autograd, the graph is that ledger. The forward pass runs the business and builds the record simultaneously.

The Chain Rule: Why Backpropagation Works

To understand why backpropagation works, you need the chain rule. Do not worry — it is simpler than it sounds.

Suppose you have a composed function: one operation feeds into another. You bake a loaf of bread. The temperature of your oven affects the colour of the crust. The colour of the crust affects the score a bread judge gives it. If you want to know how the oven temperature affects the score, you cannot simply divide the score by the temperature — the relationship passes through the intermediate quantity (crust colour). You must multiply two rates of change: how much crust colour changes per degree of oven temperature, and how much score changes per unit of crust colour.

That is the chain rule. The rate of change of the final output with respect to an early input equals the product of all the local rates of change along the path.

For a neural network, the loss is the final output and each weight is an early input. The path between them may pass through dozens of operations. The chain rule says: to find how the loss responds to a weight, multiply all the local gradients along the path from that weight to the loss.

Why Not Simple Division?

You might ask: why can we not just divide the change in loss by the change in weight? The answer is that division only works when there is a direct, linear relationship between two quantities. In a neural network, every weight is embedded in a chain of nonlinear operations — activations like ReLU or sigmoid that bend and reshape the signal. Simple division ignores these intervening transformations. The chain rule accounts for them precisely.

Think first: if a weight appears in two separate paths to the loss, what happens to its gradient? +

The gradients from both paths are added together. This follows directly from the chain rule applied to a DAG: when multiple paths connect a weight to the final loss, each path contributes its own partial gradient, and the total gradient is their sum. PyTorch handles this automatically — when .backward() is called, gradient contributions accumulate into the .grad attribute of each leaf tensor.

The Backward Pass: Tracing the Audit Trail

When you call loss.backward(), PyTorch traverses the computational graph in reverse — from the loss node back toward the leaf parameters. At each node, it calls the stored backward function, passing in the gradient that arrived from the nodes above. The backward function computes the local gradient and passes the result to the nodes below.

This process is called reverse-mode automatic differentiation. It is efficient because it computes the gradient of one scalar output (the loss) with respect to many inputs (all parameters) in a single pass. This is the right mode when you have far more parameters than outputs — which is always the case in neural networks.

Here is the traversal step by step, using the earlier example:

Start at the loss

The gradient of the loss with respect to itself is 1. This seeds the backward pass. Every other gradient is expressed relative to this starting value.

Move to the addition node

The + 1 operation does not change the gradient. Adding a constant shifts the output but does not change how sensitive it is to its inputs. Gradient passes through unchanged.

Move to the multiplication node

The * 2 operation scales the gradient by 2. Whatever gradient arrived from above is multiplied by the local derivative of the multiplication operation.

Move to the squaring node

The ** 2 operation has local derivative 2 * x. With x = 3, the local derivative is 6. The arriving gradient is multiplied by 6. The final gradient of z with respect to x is 1 * 2 * 6 = 12.

Accumulate into .grad

The computed gradient (12) is added to x.grad. If x.grad was zero before, it is now 12. This is the signal that tells the optimizer how to adjust x.

You can verify this yourself:

z.backward()
print(x.grad)   # tensor(12.)  — matches our manual calculation

The Gradient Accumulation Problem

Here is a detail that trips up many beginners, and that has an elegant explanation once you understand the graph.

Call loss.backward() twice without zeroing the gradients in between. You will find that the gradients are doubled. Call it three times and they are tripled.

This is not a bug. It is the system working as designed. PyTorch accumulates gradients into .grad rather than replacing them. The reason is that some training patterns genuinely require accumulating gradients across multiple forward passes before updating parameters — for example, when simulating a large batch size on a GPU with limited memory by running several small batches.

But in standard training, you want fresh gradients on every update. Accumulated gradients from previous steps corrupt the update direction — the optimizer sees a mixture of the current batch's signal and the previous batch's signal. This produces noisy, slow, or divergent training.

The fix is a single line before each forward pass:

optimizer.zero_grad()   # clear accumulated gradients
outputs = model(inputs)
loss = criterion(outputs, targets)
loss.backward()         # compute fresh gradients
optimizer.step()        # update parameters

The call to optimizer.zero_grad() sets every parameter's .grad to zero. The subsequent backward pass then accumulates from zero, giving you a clean gradient for that batch alone.

Active Debate

Should you call optimizer.zero_grad(set_to_none=True) instead of the default? PyTorch documentation recommends this in newer versions: setting gradients to None rather than zero avoids allocating memory for gradient tensors on parameters that did not receive gradients in that pass. The performance difference is minor for small models but measurable for very large ones. The behavior is identical for standard training loops — the optimizer checks for None the same way it checks for zero.

Detaching from the Graph

Sometimes you want to use a tensor's value in a computation without tracking that computation for backpropagation. A common case: computing a metric for logging purposes. You do not want the logging computation to add nodes to the graph or consume memory.

PyTorch provides two tools for this:

# Option 1: detach creates a new tensor with no gradient history
value = loss.detach().item()

# Option 2: no_grad context manager disables graph building entirely
with torch.no_grad():
    predictions = model(inputs)   # no graph built

The torch.no_grad() context is especially useful during inference and validation: you do not need gradients, and disabling the graph saves both memory and computation.

How the Three Major Frameworks Differ

The underlying principle is the same across PyTorch, TensorFlow, and JAX, but the implementation choices differ meaningfully.

Framework
Graph model
PyTorch
Dynamic (define-by-run). The graph is rebuilt on every forward pass. Enables arbitrary Python control flow inside the model.
TensorFlow 2 / JAX
TF2 defaults to eager (dynamic) like PyTorch, but offers @tf.function to compile static graphs. JAX uses function transformations (jit, grad) that trace a static graph on the first call and reuse it.

For learning purposes, PyTorch's dynamic graph is far more transparent. You can inspect the graph, add print statements inside the model, and reason about what is happening step by step. This is why most deep learning courses — including this one — use PyTorch as the teaching framework.

A Minimal Working Example

Here is the complete training loop for the simplest possible neural network: a single linear layer fitting y = 2x.

import torch
import torch.nn as nn

# Data: y = 2x
x = torch.tensor([[1.0], [2.0], [3.0], [4.0]])
y = torch.tensor([[2.0], [4.0], [6.0], [8.0]])

# Model: one linear layer, no activation
model = nn.Linear(in_features=1, out_features=1)
optimizer = torch.optim.SGD(model.parameters(), lr=0.01)
criterion = nn.MSELoss()

for epoch in range(100):
    optimizer.zero_grad()        # 1. Clear old gradients
    predictions = model(x)       # 2. Forward pass — graph is built here
    loss = criterion(predictions, y)   # 3. Compute loss
    loss.backward()              # 4. Backward pass — gradients computed
    optimizer.step()             # 5. Update parameters

print(model.weight)   # Should be close to [[2.0]]
print(model.bias)     # Should be close to [[0.0]]

Every line in this loop corresponds to a concept in this page. Step 1 clears accumulation. Step 2 builds the graph. Step 4 traces it backward. Step 5 uses the gradients to adjust the weights. After 100 iterations, the model has learned that the relationship between x and y is multiplication by 2.

What the Graph Looks Like During Step 2

When model(x) runs, PyTorch performs a linear transformation: output = x @ weight.T + bias. This creates two nodes in the graph — one for the matrix multiplication, one for the bias addition. Both nodes point back to weight and bias, which are leaf tensors with requires_grad=True. When criterion(predictions, y) runs, a loss node is added. The graph is now complete.

When loss.backward() runs in step 4, PyTorch walks this graph in reverse. The gradient flows backward through the MSE loss node, through the bias addition node into bias.grad, and through the matrix multiplication node into weight.grad. These two scalars — one for weight, one for bias — are the entire gradient signal that the optimizer needs.
