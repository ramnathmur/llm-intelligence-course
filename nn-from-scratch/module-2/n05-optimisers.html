Optimisers — SGD to Adam

Momentum and adaptive learning rates are the difference between training in hours and days

~15 min
Anchor Analogy
Picture a ball rolling down a hilly landscape toward the lowest valley. Plain SGD has no memory — it bounces from slope to slope, changing direction at every step based only on the current tilt beneath it. Momentum builds up speed in consistent directions, so the ball rolls smoothly through shallow valleys rather than oscillating. Adam goes further: it also checks how rough each patch of terrain is and slows down on steep, erratic surfaces while speeding up on smooth, reliable gradients. Each upgrade fixes one specific failure mode of the one before it.
Why This Matters

Plain SGD works but slowly. The progression from SGD to Momentum to RMSprop to Adam is a story of understanding why gradients misbehave and engineering fixes for each failure mode. Every large language model you have ever interacted with was trained on an Adam variant — specifically AdamW. When you open a framework and pick an optimiser from a dropdown, you are choosing a theory of how gradients should be used. Understanding the genealogy lets you diagnose training problems, tune hyperparameters with intent rather than guesswork, and know which optimiser to reach for when Adam is not the right answer.

After This Page, You Can Argue

Why the sequence SGD to Momentum to RMSprop to Adam exists, what specific failure mode each one fixes, how Adam combines the solutions from both Momentum and RMSprop into two running averages, and when you would choose plain SGD with momentum over Adam rather than defaulting to Adam in every setting.

Prerequisites: N01 — Autograd — How Gradients Flow

The Job of an Optimiser

After autograd computes gradients, something must decide how to use them. That is the optimiser's job. It takes the gradient — a direction pointing toward higher loss — and produces an update to each parameter that nudges the network toward lower loss.

The naive rule is simple: subtract the gradient, scaled by a small number called the learning rate. This is gradient descent. In practice you do it on one batch at a time, so it is called Stochastic Gradient Descent, or SGD. It works. But it has failure modes that become obvious the moment you train anything non-trivial.

The history of optimisers is a history of diagnosing those failure modes and designing targeted fixes. Each algorithm in the family tree is an answer to a specific question: why is this training loop behaving badly, and what structural change would fix it?

Stochastic Gradient Descent: The Baseline

SGD performs one operation per step: multiply the gradient by a negative learning rate and add it to the parameter.

# Plain SGD update rule
# theta: parameter, g: gradient, lr: learning rate
theta = theta - lr * g

The word "stochastic" is there because you use one mini-batch to estimate the gradient over the full dataset. That estimate is noisy. Different batches give slightly different gradients, so the path through the loss landscape is jagged rather than smooth.

SGD has two failure modes worth understanding deeply, because each one motivates a separate upgrade.

Failure Mode 1: Oscillation in Ravines

Imagine the loss landscape is shaped like a narrow canyon running diagonally. The walls of the canyon are steep. The floor of the canyon descends gently toward the minimum.

SGD sees steep walls and takes large steps perpendicular to the canyon floor — bouncing left and right across the canyon. Meanwhile it takes tiny steps along the canyon floor because the gradient there is small. Progress toward the minimum is slow. The path zigs and zags instead of running straight down the canyon.

This is the oscillation problem. The gradient at each step points in the locally steepest direction, which is often across the canyon rather than down it. SGD has no memory of where it has been, so it cannot recognise that it keeps undoing its own progress.

Failure Mode 2: Uniform Treatment of All Parameters

A neural network processes vastly different features. Some features appear in almost every training example — they receive a gradient signal on every batch. Other features are rare — a colour described by a single word, a domain-specific term that appears in 0.1% of the training data. Those rare features receive an update almost never.

SGD applies the same learning rate to both. The common features get updated too aggressively (causing oscillation) while the rare features barely move at all. This mismatch slows convergence across the board.

Momentum: Fixing Oscillation

Momentum fixes the oscillation problem by giving the optimiser a memory of past gradients. Instead of using the raw gradient at each step, it maintains a running average of all past gradients and moves in that averaged direction (Polyak, 1964).

# Momentum update rule
# v: velocity (running average of gradients)
# beta: momentum coefficient, typically 0.9
v = beta * v + (1 - beta) * g
theta = theta - lr * v

The key number is beta, typically set to 0.9. At each step, 90% of the velocity carries forward from the previous step, and 10% comes from the new gradient. The effect is a weighted average of all past gradients, with recent ones weighted more heavily than older ones.

Why This Fixes Oscillation

Return to the canyon analogy. In the perpendicular direction (across the canyon), gradients alternate in sign: one step pushes left, the next pushes right. When you average them, they cancel each other out. Momentum suppresses oscillation in the perpendicular direction automatically.

Along the canyon floor, gradients consistently point the same way: toward the minimum. Averaging them does not cancel them — it amplifies them. Momentum builds up speed in the consistent direction while dampening the noisy perpendicular direction. The ball rolls down the canyon floor instead of bouncing across its walls.

Key Insight

Momentum works because it exploits signal-to-noise ratio. Directions that are consistently useful accumulate in the velocity and grow. Directions that are noisy and inconsistent cancel in the velocity and shrink. This is why momentum dramatically improves convergence even with no change to the learning rate.

Choosing the Beta Coefficient

Beta controls how much history the velocity carries. At beta = 0.9, the effective window is approximately 10 recent gradients. At beta = 0.99, it is approximately 100. Higher beta means smoother trajectories but slower response to genuine changes in the loss landscape.

In practice, 0.9 is the default that works across most settings. Tuning beta is rarely the right first step when training is struggling — the learning rate and architecture choices usually matter more.

RMSprop: Fixing the Uniform Learning Rate

Momentum solved oscillation. It did not solve the second failure mode: the fact that rare features receive too little update signal relative to common ones.

Geoffrey Hinton proposed RMSprop as a fix in an unpublished Coursera lecture in 2012 (Hinton, 2012). The idea is to track how large the gradient for each parameter has been recently, and use that to scale the update. Parameters with consistently large gradients get a smaller effective learning rate. Parameters with small gradients get a larger effective learning rate.

# RMSprop update rule
# s: running average of squared gradients (one per parameter)
# epsilon: small constant to prevent division by zero
s = beta * s + (1 - beta) * g**2
theta = theta - lr * g / (sqrt(s) + epsilon)

Notice s accumulates squared gradients. The square root of s approximates the root mean square of recent gradients for that parameter. Dividing the gradient by this value normalises the update: large-gradient parameters get divided by a large number, shrinking their update; small-gradient parameters get divided by a small number, preserving their update.

The Consulting Parallel

Think of a consulting team where some team members speak at every meeting and others rarely contribute. A flat performance system treats every comment equally. A smart manager weights quiet members' rare contributions more heavily, recognising that frequency of input does not equal importance of input. RMSprop applies the same logic: it discounts parameters that shout frequently and amplifies parameters that speak rarely.

What RMSprop Does Not Fix

RMSprop adapts the learning rate per parameter. It does not add momentum. Training on RMSprop is smoother than plain SGD for heterogeneous data, but the trajectory can still oscillate in ravines because there is no velocity accumulation. You get one fix without the other.

Adam: Combining Both Fixes

Adam — Adaptive Moment Estimation — combines momentum and RMSprop into a single algorithm (Kingma & Ba, 2015). This is why it uses two running averages rather than one.

  • First moment (m): a running average of the gradients, just like momentum. This is the direction to move.
  • Second moment (v): a running average of the squared gradients, just like RMSprop. This is the scale of the terrain in each direction.
# Adam update rule
# m: first moment (velocity), v: second moment (squared gradient average)
# beta1: typically 0.9,  beta2: typically 0.999,  epsilon: 1e-8
m = beta1 * m + (1 - beta1) * g
v = beta2 * v + (1 - beta2) * g**2

# Bias correction (explained below)
m_hat = m / (1 - beta1**t)
v_hat = v / (1 - beta2**t)

theta = theta - lr * m_hat / (sqrt(v_hat) + epsilon)

The final update divides the momentum-corrected gradient direction by the RMSprop-style scale factor. In the canyon analogy: Adam builds up speed down the canyon floor (momentum) while also moving faster on smooth terrain than rough terrain (adaptive rate).

Bias Correction: Why It Exists

Both running averages start at zero. Early in training, when t is small, this zero initialisation pulls the estimates toward zero — they are biased downward. The first few steps look like the gradients are near-zero even when they are not.

Kingma and Ba noticed this and added a correction: divide each running average by (1 - beta**t). When t is small, the denominator is much less than 1, which scales the estimate back up toward its true value. As t grows large, the denominator approaches 1, and the correction term disappears. This is a simple but important stabilisation for early training.

The Two Beta Hyperparameters

Adam has two beta values. Beta1 (default 0.9) controls the momentum window — the same as standalone momentum. Beta2 (default 0.999) controls the second moment window. The high default for beta2 means Adam tracks squared gradients over a very long window — approximately 1,000 recent steps. This makes the adaptive scaling stable and slow to change, which prevents the effective learning rate from collapsing too aggressively in early training.

Think first: Why does Adam use a much larger beta for the second moment (0.999) than the first moment (0.9)? +

The first moment (gradient direction) can and should respond quickly to the current landscape. A beta of 0.9 means recent gradients dominate — momentum builds fast. The second moment (scaling factor) should be stable and slow to change. If the scaling estimate oscillated wildly, the effective learning rate per parameter would be noisy, destabilising training. A beta of 0.999 averages squared gradients over approximately 1,000 steps, which produces a smooth, reliable estimate of how "rough" each parameter's terrain is. The asymmetry is intentional: fast adaptation of direction, slow adaptation of scale.

AdamW: The Version That Trains LLMs

The version of Adam actually used to train LLMs is AdamW, proposed by Loshchilov and Hutter (Loshchilov & Hutter, 2019). The W stands for weight decay.

Weight decay is a regularisation technique: you add a small penalty to large weights during training, nudging the model toward smaller, simpler parameters. In standard Adam, weight decay is implemented as L2 regularisation — it is added to the gradient before the adaptive scaling step. This means the effective penalty changes depending on how the second moment scales each parameter. Large-gradient parameters receive a smaller weight decay penalty after scaling, which is not the intended behaviour.

AdamW decouples weight decay from the gradient update entirely. The weight decay term is applied directly to the parameter at a fixed rate, independent of the adaptive scaling. The gradient update and the regularisation term operate separately.

# AdamW: weight decay applied directly, not through gradient
# (Adam update as above, then:)
theta = theta - lr * (m_hat / (sqrt(v_hat) + epsilon) + weight_decay * theta)

In practice, AdamW trains more effectively than Adam on large models, especially with high learning rates. When you call torch.optim.AdamW, this decoupled weight decay is what you are getting.

When to Use SGD with Momentum Instead

Adam is not always the right choice. SGD with momentum has a well-documented advantage in computer vision tasks — particularly image classification with convolutional networks. Several papers have shown that SGD with momentum achieves better final test accuracy than Adam on CIFAR and ImageNet benchmarks, even though Adam converges faster early in training (Wilson et al., 2017).

The intuition is that Adam's adaptive learning rates can over-specialise to the training data. By scaling down updates in directions where gradients are large and consistent, Adam may be implicitly reducing the update signal for the most important features. SGD with momentum does not make this adjustment and can generalise better in some settings.

Practical Rule of Thumb

Default to AdamW for language models, transformers, and any setting where you are unsure. Use SGD with momentum when training convolutional networks for vision tasks where you have the compute budget to tune a learning rate schedule carefully. If your training budget is tight and you cannot afford a long hyperparameter search, Adam is the safer default because it is less sensitive to learning rate choice.

Learning Rate Schedules: The Missing Piece

No optimiser specification is complete without a learning rate schedule. The learning rate in the update rules above is not a fixed constant in practice — it decays over training.

The most common schedule for transformer training is cosine annealing: the learning rate starts high, then follows a cosine curve down to near zero by the end of training. The intuition is that large steps early in training help the network escape poor regions of the loss landscape quickly, while small steps late in training allow fine-grained convergence to a better local minimum.

import torch
model = ...  # your neural network
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3, weight_decay=0.01)
scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=num_epochs)

for epoch in range(num_epochs):
    train_one_epoch(model, optimizer)
    scheduler.step()  # decay the learning rate

Warmup is another common addition: the learning rate starts from near zero, increases linearly for the first few thousand steps, and then follows the cosine decay. This prevents large early updates from pushing the model into a bad region before the gradient estimates have stabilised.

Comparing the Optimisers Side by Side

Update rule: theta = theta - lr * g

Memory: None. Each step depends only on the current gradient.

Fixes: Nothing beyond basic gradient descent.

Failure modes: Oscillation in ravines; uniform learning rate across all parameters.

When to use: Almost never without momentum. Useful as a teaching baseline.

Update rule: v = beta * v + (1 - beta) * g; theta = theta - lr * v

Memory: One running average per parameter (velocity).

Fixes: Oscillation in ravines by averaging out noisy gradient directions.

Failure modes: Still uses a uniform learning rate across all parameters.

When to use: As part of SGD with momentum for convolutional vision networks; rarely on its own for language tasks.

Update rule: s = beta * s + (1 - beta) * g**2; theta = theta - lr * g / (sqrt(s) + eps)

Memory: One running average per parameter (squared gradients).

Fixes: Uniform learning rate by scaling updates per parameter based on gradient history.

Failure modes: No momentum; can still oscillate in ravines.

When to use: Recurrent networks (RNNs/LSTMs) where gradient magnitudes vary sharply; less common now that Adam is available.

Update rule: First moment (velocity) + second moment (adaptive scale) + bias correction.

Memory: Two running averages per parameter: gradient direction and squared gradient magnitude.

Fixes: Both oscillation (via first moment) and uniform learning rate (via second moment).

Failure modes: Can generalise worse than SGD with momentum on some vision benchmarks; Adam's weight decay is subtly incorrect (use AdamW).

When to use: Default for transformers, language models, and most modern architectures. Use AdamW, not plain Adam.

Key Insight

Adam is not one clever idea — it is two targeted fixes combined. Momentum fixes oscillation by averaging gradient directions over time. RMSprop fixes uneven parameter treatment by normalising updates by gradient magnitude history. Adam runs both simultaneously, maintaining two running averages per parameter. AdamW then separates weight decay from the gradient to fix a subtle regularisation bug. Understanding each step in this chain means you can diagnose which fix you need when Adam itself is misbehaving.
